# Logseq 지식 관리

## 저장소

- `~/Documents/logseq/` — 단일 지식 저장소 (Logseq graph, git 관리)
- 프로젝트 무관 범용

## 읽기/검색

- 이전 이슈/결정 참조 시 `qmd search "키워드"` 또는 `qmd query "질문"` 사용
  - 키워드 명확 → `qmd search` (BM25, 빠르고 정확)
  - 맥락적/시맨틱 → `qmd query` (Hybrid, 리랭킹)
  - 프로젝트 필터 → 프로젝트명만으로 검색 (`pj-`는 하이픈 토크나이징 실패)
- `/qmd` 스킬 활용 가능
- Logseq 파일 직접 Read도 자유
- 사용자가 과거 맥락을 참조하면 ("전에", "기존에", "아까") 먼저 `qmd search`로 기존 기록을 찾는다

## 쓰기

- **Logseq는 사용자의 개인 지식 저장소 — 쓰기 전 반드시 사용자 확인**
- **작성 시 `/logseq-write` skill을 호출**하여 규격에 맞게 작성한다
- 다음 상황에서 Logseq 기록을 능동적으로 제안한다:
  - 트러블슈팅 완료 시 → `troubleshoot/` 제안
  - 중요 결정이 확정되었을 때 → `decision/` 제안
  - QA 완료 시 → `qa/` 제안
  - 회의/슬랙 대화에서 스펙이 도출되었을 때 → `spec/` 제안
  - 배포 실패/인프라 이슈 해결 시 → `incident/` 제안

## `#` vs `[[]]` 사용 원칙

- **기본: `[[링크]]`를 사용한다** — 단어 경계가 명확하고 오작동이 없다
- **`#태그` 허용**: 저널 TODO 행의 프로젝트 태그 (`#pj-{name}`)에 한정
- **`#숫자` 금지**: `#288` → Logseq가 `[[288]]` 페이지 생성. `PR 288` 형태로 작성

## Claude의 역할

- 세션 시작 시 오래된 TODO (2주+) 발견하면 알림
- open status 페이지가 방치되었으면 리뷰 제안
- 새 TODO 작성 전 중복 확인 (`qmd search`)

## 그래프 유지보수

- 인덱스 갱신: 페이지 추가/수정 후 `qmd update && qmd embed` 실행
- open status 페이지 주기적 확인 — 해결된 것은 status 갱신
- 고립 페이지 발견 시 관련 페이지와 `[[링크]]` 연결 추가
