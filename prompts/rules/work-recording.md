# 작업 기록 규칙

## 능동적 기록 트리거

| 조건 | 행동 |
|------|------|
| 세션 내 git commit 3회+ | `/debrief` 제안 |
| PR 생성 | `/debrief` 제안 |
| **코드 작업 반영 완료** | 당일 저널에 작업 결과 기록 제안 (`- [ ]` → `- [x]` 전환, 요약 추가). namespace 페이지 생성이 적절하면 함께 제안 |
| 슬랙/이메일 내용 붙여넣기 | 관련 issue 페이지 업데이트 제안 (없으면 생성 제안) |
| open issue 페이지 Read | `last-verified::` 갱신 제안 |
| 외부 의존성 있는 작업 시작 | issue 페이지 생성 제안 |
| **사용자가 세션 종료 신호** | 당일 저널에 `- [x] {작업 요약}` **기록한다**. `/debrief` 제안 |
| **스킬/룰/훅 등 하네스 수정** | 당일 저널에 `- [x] {변경 요약}` **기록한다** |

세션 종료 신호: "여기서 종료", "끝", "정리해줘", "오늘 여기까지", "debrief" 등

## issue vs troubleshoot 판단

- 한 번의 조사→해결 → `troubleshoot`
- 외부 의존, 커뮤니케이션 반복, 주 단위 추적 → `issue`
- troubleshoot이 해결 없이 길어지면 issue로 승격

## 커뮤니케이션 기록 원칙

단순히 "메일 보냄"이 아니라 3요소 기록:
1. **무엇을 물었/요청했는지**
2. **어떤 답이 왔는지** (또는 대기 중)
3. **상태가 어떻게 바뀌었는지**

## 저널 갱신 패턴

세션 종료 시 당일 저널에 기록:
- 해당 세션의 기존 `- [ ]` / `- [/]` → `- [x]`로 변경
- 세션 요약 1줄 추가 (없으면 새 항목): `- [x] {작업 요약} #pj-{name}`
- namespace 페이지가 생성되었으면 하위에 `-> [[링크]]` 추가

> Obsidian markdown checkbox 표준 사용. 키워드 `TODO`/`DOING`/`DONE`은 logseq 잔재로 deprecated. 기존 페이지는 `ruby scripts/migrate-todo-to-checkbox.rb`로 변환됨.
>
> | 의미 | 표기 |
> |------|------|
> | 미완료 | `- [ ]` |
> | 진행 중 | `- [/]` |
> | 완료 | `- [x]` |
> | 취소 | `- [-]` |

## 진행 중 계획 갱신 (Living Plan)

저널 `- [ ]` / `- [/]`에 `# 작업 계획`이 있는 항목은 living document로 관리한다.

### 구조

```
- [/] {제목} #pj-{name}
    - # 변경 이력          ← append-only, 시간순
        - MM/DD HH:mm — {무엇이 바뀌었는지 한 줄}
    - # 현재 상태          ← 항상 최신 스냅샷으로 덮어쓴다
    - # 작업 계획          ← `- [ ]` / `- [x]` 항목, 완료 시 전환
    - # 확인 필요          ← 미결 질문, 해소되면 제거
```

### 갱신 규칙

- **현재 상태 / 작업 계획**: 최신 기준으로 덮어쓴다 (스냅샷)
- **변경 이력**: 절대 수정/삭제하지 않는다. 갱신할 때마다 한 줄 append
- **변경 이력 포맷**: `{MM/DD HH:mm} — {변경 내용 한 줄}`

### 갱신 트리거

| 이벤트 | 행동 |
|--------|------|
| 외부 스펙/산출물 갱신 (새 버전 수령) | 변경 이력 추가 + 현재 상태·작업 계획 반영 |
| 결정 확정 (미결 → 확정) | 변경 이력 추가 + 확인 필요에서 제거 |
| 스코프 변경 (항목 추가/제거) | 변경 이력 추가 + 작업 계획 갱신 |
| 새 도메인 개념 발견 | `[[링크]]`로 감싸고, 페이지 없으면 개념 페이지 생성 제안 |

### 적용 기준

모든 체크박스에 적용하는 것이 아니라, **다단계 + 외부 의존성이 있는 작업**에만 적용:
- 작업 계획이 3개 이상의 체크박스를 포함
- 외부 산출물(스펙, 디자인, 샘플)에 의존
- 여러 세션에 걸쳐 진행

단순 1회성 체크박스에는 불필요.

## freshness 운용

- `last-verified::` — 내용이 마지막으로 정확하다고 확인된 날짜
- `next-check::` — 다음 확인 예정일 (기본: 7일 후)
- 일일 wakeup에서 기한 초과 자동 감지
