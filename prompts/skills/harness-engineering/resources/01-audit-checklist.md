# Harness Audit Checklist

7차원 진단의 상세 체크리스트.

## 1. Navigation (0-3)

- [ ] 코드베이스 지도(ARCHITECTURE.md 또는 rules/architecture.md)가 존재하는가
- [ ] 패키지/모듈 구조와 각 역할이 명시되어 있는가
- [ ] 모듈 간 관계/의존 방향을 보여주는가 (디렉토리 나열이 아닌)
- [ ] 아키텍처 불변식(absences)이 명시되어 있는가 (e.g., "X는 Y에 의존하지 않는다")
- [ ] 횡단 관심사(에러 처리, 인증, 로깅 등) 섹션이 있는가
- [ ] 검색 가능한 심볼 이름으로 핵심 타입/파일을 참조하는가 (직접 링크 대신)
- [ ] BE ↔ FE 대응 매핑이 있는가
- [ ] 핵심 파일 경로(테스트 베이스, 설정 파일, ArchUnit 등)가 명시되어 있는가
- [ ] 도메인 간 흐름(이벤트 기반 포함)이 설명되어 있는가
- [ ] 에이전트가 "X 기능 수정해줘" 요청 시 탐색 없이 직행할 수 있는가

| 점수 | 기준 |
|------|------|
| 0 | 지도 없음. 매 세션 Glob/Grep 10회+ 소모 |
| 1 | CLAUDE.md에 일부 구조 언급이 있지만 체계적이지 않음 |
| 2 | Codemap(모듈 역할 + 관계/의존 방향) + 핵심 파일 경로가 문서화됨 |
| 3 | Codemap + 불변식 + 횡단 관심사 + 검색 가능 심볼. 1회 Read로 탐색 불필요 |

## 2. Knowledge Base (0-3)

- [ ] docs/ 또는 동등한 지식 저장소가 리포지터리에 존재하는가
- [ ] 설계 문서(design docs)가 리포지터리에 인코딩되어 있는가
- [ ] 제품 스펙이 리포지터리에서 참조 가능한가
- [ ] 실행 계획(exec plans)이 리포지터리에서 관리되는가 (선택)
- [ ] 외부에만 있는 지식(Slack, Google Docs)이 남아있지 않은가
- [ ] "에이전트가 접근 못 하면 없는 것" 원칙이 지켜지는가

| 점수 | 기준 |
|------|------|
| 0 | 지식이 외부(Slack, Google Docs, 사람 머릿속)에만 존재 |
| 1 | 일부 문서가 리포지터리에 있지만 구조화되지 않음 |
| 2 | docs/ 구조 존재, 주요 설계/스펙이 리포지터리에 인코딩 |
| 3 | 구조화된 지식 베이스 + 외부 참조(llms.txt) + 문서 신선도 관리 |

## 3. Instructions (0-3)

- [ ] CLAUDE.md가 존재하는가
- [ ] CLAUDE.md가 목차 역할인가 (≤100줄, 상세는 rules/docs로 분리)
- [ ] 불필요한 정보가 포함되어 있지 않은가 — "제거해도 에이전트가 실수할까?"
- [ ] 코드에서 유추 가능한 것이 포함되어 있지 않은가
- [ ] 린터가 강제하는 스타일을 instruction에 중복 작성하지 않았는가
- [ ] ARCHITECTURE.md 참조가 CLAUDE.md에 포함되어 있는가
- [ ] progressive disclosure가 적용되어 있는가

| 점수 | 기준 |
|------|------|
| 0 | CLAUDE.md 없음 |
| 1 | CLAUDE.md 있으나 백과사전화 (300줄+, 모든 규칙 나열) |
| 2 | 계층 구조 + rules 분리, ARCHITECTURE 참조 |
| 3 | 목차(≤100줄) + progressive disclosure + ARCHITECTURE/docs 참조 |

## 4. Constraints (0-3)

- [ ] 코딩 컨벤션이 rules로 명문화되어 있는가
- [ ] 각 rule에 '이유(Why)'가 포함되어 있는가
- [ ] 올바른/잘못된 예시가 포함되어 있는가
- [ ] 사용자가 반복 교정하는 패턴이 rule로 없는가
- [ ] 아키텍처 규칙이 테스트 코드로 강제되는가 (의존 방향, 레이어 경계 등)
- [ ] 아키텍처 테스트가 CI에서 실행되는가

| 점수 | 기준 |
|------|------|
| 0 | 명문화된 규칙 없음 |
| 1 | CLAUDE.md에 규칙이 섞여 있음 |
| 2 | rules/*.md로 분리, Why 포함 |
| 3 | 자동 검증까지 연동 — 아키텍처 테스트(ArchUnit/lint)가 CI에서 의존 방향·레이어 경계를 강제 |

## 5. Enforcement (0-3)

- [ ] pre-commit hook이 있는가
- [ ] pre-push hook이 있는가
- [ ] Claude Code hooks가 설정되어 있는가 (PreToolUse, Stop 등)
- [ ] 보호 파일(migration, secrets)에 대한 gate가 있는가
- [ ] 커스텀 린트 에러 메시지가 에이전트에게 수정 지침을 제공하는가

| 점수 | 기준 |
|------|------|
| 0 | git hooks 없음, Claude hooks 없음 |
| 1 | git hooks만 있음 |
| 2 | git hooks + Claude hooks + 커스텀 린터 |
| 3 | 린트 에러가 수정 지침을 주입 + 프로젝트 특화 gates |

## 6. Memory (0-3)

- [ ] MEMORY.md가 존재하는가
- [ ] 200줄 이내인가
- [ ] 중복 항목이 없는가
- [ ] obsolete 항목이 없는가 (이미 rules로 승격된 것 등)
- [ ] 메모리 유형이 적절히 분류되어 있는가

| 점수 | 기준 |
|------|------|
| 0 | MEMORY.md 없음 |
| 1 | 있으나 정리되지 않음 |
| 2 | 정리됨, 200줄 이내 |
| 3 | 유형별 분류, 주기적 정리, 인덱스 최신 |

## 7. Entropy Management (0-3)

- [ ] ARCHITECTURE.md가 실제 코드와 일치하는가 (새 패키지 누락, 삭제된 패키지 잔존)
- [ ] rules가 현재 코드 관행과 일치하는가
- [ ] CLAUDE.md에 오래된 정보가 있는가
- [ ] 기술 부채를 추적하고 점진적으로 상환하는 프로세스가 있는가
- [ ] 문서 신선도를 검증하는 메커니즘이 있는가 (CI, 정기 리뷰 등)

| 점수 | 기준 |
|------|------|
| 0 | 한 번 쓰고 방치. 문서와 코드가 이미 불일치 |
| 1 | 간헐적으로 사람이 수동 정리 |
| 2 | 정기 리뷰 프로세스 존재 (월 1회 등) |
| 3 | 자동화된 신선도 검증 + 정기 정리 PR + 기술 부채 추적 |
