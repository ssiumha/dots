# Context Engineering 전략

에이전트가 접근할 수 없는 것은 사실상 존재하지 않는다.
리포지터리를 에이전트의 유일한 기록 시스템으로 만드는 것이 context engineering의 목표.

## 5대 전략

### 1. Selection (선택)

관련 컨텍스트만 선택하고 나머지는 제외한다.

- CLAUDE.md에 "모든 것"을 넣지 않는다 — 목차 역할 (~100줄)
- 에이전트가 스스로 발견할 수 있는 것은 제외 (디렉토리 구조, 코드베이스 개요)
- 에이전트가 추측할 수 없는 것만 포함 (비표준 명령, 아키텍처 결정, 환경 quirks)

### 2. Compression (압축)

정보 밀도를 최적화한다.

- 코드 스니펫 대신 `파일:라인` 포인터
- 산문보다 테이블/리스트
- `/llms.txt` > 전체 HTML 문서
- Sub-agent 응답은 `filepath:line` 형태로 압축

### 3. Ordering (배치)

중요한 정보를 앞에 배치한다.

- CLAUDE.md 상단에 ARCHITECTURE.md 참조
- 자주 참조되는 명령을 먼저
- 예외/edge case는 rules 파일로 분리

### 4. Isolation (격리)

Sub-agent로 컨텍스트를 격리한다.

- 탐색/조사 작업은 sub-agent에 위임 → 메인 컨텍스트 보호
- 각 sub-agent에 focused된 프롬프트와 제한된 도구
- 결과만 메인으로 반환 (전체 탐색 과정은 버림)

### 5. Format (형식)

모델이 처리하기 좋은 형식을 사용한다.

- 구조화된 데이터: JSON > 비정형 텍스트
- 기능 목록: JSON 배열 (Markdown보다 모델이 부적절하게 수정할 가능성 낮음)
- YAML frontmatter로 glob 패턴 지정 → 특정 파일 타입에만 rule 적용

## CLAUDE.md 작성 Best Practices

### 분량 기준

- **목차 역할**: ~100줄. 상세는 rules/docs로 분리
- **"이 줄을 제거하면 에이전트가 실수할까?"** → 아니면 삭제

### 포함해야 할 것

| 포함 | 이유 |
|------|------|
| ARCHITECTURE.md 참조 | 코드베이스 지도로 직행 |
| 에이전트가 추측할 수 없는 Bash 명령 | `just gen-api` 같은 프로젝트 고유 명령 |
| 기본값과 다른 코드 스타일 규칙 | 표준과 다른 컨벤션만 |
| 테스트 실행 방법 | 프로젝트별로 다름 |
| 아키텍처 결정 | 코드만 봐서는 "왜"를 알 수 없음 |
| 개발 환경 quirks | 비표준 포트, 특이한 설정 |

### 제외해야 할 것

| 제외 | 이유 |
|------|------|
| 코드를 읽으면 알 수 있는 것 | 에이전트가 스스로 발견 |
| 표준 언어 컨벤션 | 모델이 이미 알고 있음 |
| 상세 API 문서 | docs/references/ 또는 링크로 대체 |
| 코드 스니펫 | 오래되면 혼란, 파일:라인 포인터가 나음 |
| 파일별 코드베이스 설명 | ARCHITECTURE.md에서 커버 |
| 린터가 강제하는 스타일 | "린터가 할 일은 린터에게" |

## docs/ — 리포지터리를 기록 시스템으로

Slack, Google Docs, 사람 머릿속에 있는 지식은 에이전트에게 없는 것이다.
리포지터리의 docs/를 기록 시스템으로 취급한다.

### 권장 구조

```
docs/
├── design-docs/          설계 문서 + index.md
├── exec-plans/           실행 계획 (active/, completed/)
├── product-specs/        제품 스펙
├── references/           외부 참조 (llms.txt 등)
├── generated/            자동 생성 (db-schema.md 등)
└── tech-debt-tracker.md  기술 부채 추적 (선택)
```

### 원칙

- **계획은 일급 아티팩트**: 복잡한 작업의 실행 계획은 진행 상황, 의사결정 로그와 함께 리포지터리에 저장
- **점진적 공개**: 에이전트는 CLAUDE.md(목차) → ARCHITECTURE.md(지도) → docs/(상세) 순서로 깊이 탐색
- **문서 신선도 관리**: 정기 gardening으로 코드와 문서의 불일치를 점검하고 갱신

## Level 시스템 (하네스 성숙도)

### Level 1: 개인 (1-2시간)

- CLAUDE.md에 프로젝트 컨벤션 작성
- Pre-commit hooks로 린팅/포맷팅
- 자기 검증용 테스트 스위트
- 일관된 디렉토리 네이밍

### Level 2: 소규모 팀 (1-2일)

- Level 1 + 팀 컨벤션
- ARCHITECTURE.md + rules/ 분리
- CI로 아키텍처 제약 강제
- 공유 프롬프트 템플릿 / rules
- 에이전트 전용 리뷰 체크리스트

### Level 3: 프로덕션 (1-2주)

- Level 2 + docs/ 지식 베이스
- 커스텀 린터 (에러 메시지 = 에이전트 수정 지침)
- 관찰 가능성 통합
- 스케줄된 엔트로피 관리 (정기 gardening)
- 에스컬레이션 정책

## Hooks Exit Code 컨벤션

| Exit Code | 의미 | 동작 |
|-----------|------|------|
| 0 | 성공 | 침묵 — 컨텍스트에 아무것도 전달 안 함 |
| 2 | 실패 | stderr만 에이전트에게 표면화 |

**"성공은 침묵, 실패만 verbose"** — Hook이 성공할 때마다 출력을 컨텍스트에 넣으면 토큰 낭비.

## Sub-agent 활용 패턴

### 적합한 용도

- 특정 정의/구현 탐색
- 코드베이스 패턴 분석
- 서비스 경계 간 정보 흐름 추적
- 문서/코드/웹 리서치

### 출력 형식

Sub-agent는 압축된 응답을 반환해야 한다:
- `filepath:line` 형태의 위치 정보
- 핵심 발견 사항 1-3줄 요약
- 전체 탐색 과정이나 코드 내용은 반환하지 않음

### 비용 최적화

- Parent = 고성능 모델 (판단/계획)
- Sub-agent = 경량 모델 (탐색/검색)

> 2+ 에이전트 협업이 필요한 경우(Agent Teams)는 `resources/08-agent-team-patterns.md` 참조.
> Sub-agent = 컨텍스트 격리, Agent Team = 오케스트레이션 — 관심사가 다르다.
