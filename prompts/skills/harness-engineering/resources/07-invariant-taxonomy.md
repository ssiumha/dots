# 불변식 분류 체계 (Invariant Taxonomy)

불변식의 전체 유형을 분류하고, 발견 방법론과 우선순위 프레임워크를 제공한다.
ARCHITECTURE.md에 불변식을 어떻게 작성하는지는 `resources/04-architecture-guide.md`, 어떻게 기계적으로 강제하는지는 `resources/06-arch-test-tools.md` 참조.

## 불변식이란

"깨지면 시스템에 실질적 문제가 생기는" 제약. **"존재하지 않는 것"**으로 표현되며, 코드를 읽어서는 발견할 수 없다 — "하지 않는 것"은 코드에 없기 때문이다.

> 의존성은 변경이 전파되는 방향이다. A가 B에 의존하면 B의 변경이 A에 영향을 준다.
> 의존 방향을 제어하지 않으면 한 모듈의 변경이 예측 불가능하게 시스템 전체로 퍼진다.

**판단 기준**: "이 규칙이 깨지면 시스템에 실질적 문제가 생기는가?" — Yes면 불변식, No면 불필요.

## 분류

| 카테고리 | 정의 | 위반 시 문제 |
|----------|------|-------------|
| **구조 (Structural)** | 의존 방향, 레이어 경계, 순환 금지, 모듈 독립성 | 변경 전파가 예측 불가. 한 모듈 수정이 시스템 전체로 파급 |
| **데이터 무결성 (Data Integrity)** | 경계 파싱, 스키마 호환, migration 순서, nullable 규칙 | 잘못된 데이터가 시스템 깊숙이 침투. 디버깅 비용 기하급수적 |
| **API 계약 (API Contract)** | 하위 호환, 버전 정책, 응답 형태 안정성, 에러 코드 체계 | 외부 소비자(클라이언트, 파트너) 장애. 신뢰 손실 |
| **보안 (Security)** | 인증/인가 경계, secret 관리, 입력 검증 위치, 신뢰 경계 | 데이터 유출, 권한 상승, 규제 위반 |
| **취향 (Taste)** | 로깅 형식, 네이밍 컨벤션, 파일 크기, 에러 처리 패턴 통일 | 코드 일관성 점진적 저하. 에이전트가 나쁜 패턴을 복제 |
| **운영 (Operational)** | 성능 SLA, 배포 불변식, 옵저버빌리티 요구, 리소스 제한 | SLA 위반, 장애 탐지 지연, 운영 비용 증가 |

### 카테고리별 구체 예시

#### 구조 (Structural)

- UI → Entity 직접 참조 금지. 반드시 Service/DTO를 통해서만 접근
- Feature A ↛ Feature B — 같은 레벨 모듈은 서로 독립
- 고정 레이어: Types → Config → Repo → Service → Runtime → UI (역방향 금지)
- 교차 관심사는 명시적 인터페이스(Providers)를 통해서만 유입

#### 데이터 무결성 (Data Integrity)

- 모든 외부 데이터는 ingestion 경계에서 파싱/검증. 하류 모듈은 raw 데이터를 직접 처리하지 않음
- DB migration은 append-only. 기존 migration 파일 수정 금지
- nullable 필드는 도메인 경계에서 명시적으로 핸들링. Optional 전파 금지
- 도메인 모델의 시간 필드는 타임존 인식 타입만 허용 — 타임존 없는 타입은 앱/DB 서버 설정에 따라 해석이 달라짐 (Java: `Instant`, Python: `datetime(tzinfo=...)`)
- Enum 직렬화는 문자열 기반 강제 — 순서 기반(ordinal)은 enum 재정렬 시 기존 저장 데이터와 불일치
- 도메인 모델은 공통 베이스 타입 상속 — 감사 필드(id, createdAt, updatedAt)의 일관성 보장. 누락 시 추적 불가

#### API 계약 (API Contract)

- 공개 API의 기존 필드 삭제/이름 변경 금지. deprecation → 새 필드 추가 → 제거의 3단계
- 에러 응답은 통일된 형태(code, message, details). 자유 형식 에러 금지
- 버전 변경 없이 breaking change 금지

#### 보안 (Security)

- 인증 미들웨어 없는 엔드포인트는 allowlist에 명시적 등록 필수 (/health, /metrics 등)
- Secret은 환경변수/시크릿 매니저를 통해서만 주입. 코드/설정 파일에 하드코딩 금지
- 사용자 입력은 신뢰 경계(controller/handler)에서 검증. 도메인 내부에서 raw 입력 처리 금지

#### 취향 (Taste)

- 구조화된 JSON 로깅만 허용. `console.log`/`print` 직접 사용 금지
- 네이밍 규칙: 도메인 모델은 PascalCase, 유틸리티는 camelCase 등 팀 합의 규칙
- 단일 파일 300줄 초과 시 분리 필요

#### 운영 (Operational)

- API 응답 시간 800ms 이내. 초과 시 비동기 처리로 전환
- 모든 엔드포인트에 분산 트레이싱 적용. trace_id 없는 요청 금지
- 배포는 무중단(zero-downtime). DB 스키마 변경은 backward compatible만 허용

## 발견 방법론 (Discovery)

불변식은 "하지 않는 것"이므로 능동적으로 발견해야 한다.

| 기법 | 방법 | 발견하는 유형 |
|------|------|---------------|
| **장애 역추적** | 인시던트/포스트모템에서 "이걸 안 했으면 발생하지 않았을 것" 추출 | 보안, 데이터 무결성, 운영 |
| **리뷰 반복 코멘트** | PR 리뷰에서 반복 지적되는 패턴 수집 — 3회+ 반복이면 불변식 후보 | 취향, 구조 |
| **레이어 경계 워크스루** | 고정 레이어(Types→Config→...→UI) 따라 각 경계에서 "넘어가면 안 되는 것" 도출 | 구조 |
| **신입 질문 수집** | "이거 왜 이렇게 해요?" = 문서화되지 않은 불변식 | 전 유형 |
| **Absence 스캔** | "X는 Y를 하지 않는다" 형태로 각 모듈 쌍을 체계적 순회 | 구조, 데이터 무결성 |
| **의존성 그래프 시각화** | 실제 그래프에서 의도치 않은 연결 발견. 있어선 안 될 간선 = 불변식 후보 | 구조 |
| **데이터 흐름 추적** | 외부 입력이 어디서 파싱/검증되는지 추적. YOLO style(추측 기반 처리) 탐지 | 데이터 무결성, 보안 |

## 우선순위 프레임워크

어떤 불변식부터 강제할 것인가:

| 우선순위 | 기준 | 예시 카테고리 |
|----------|------|---------------|
| **P0** | 위반 시 데이터 손실/보안 사고 | 보안, 데이터 무결성 |
| **P1** | 위반 시 아키텍처 부패 (복구 비용 기하급수적 증가) | 구조 |
| **P2** | 위반 시 외부 소비자에게 영향 | API 계약 |
| **P3** | 위반 시 운영 비용 증가 | 운영 |
| **P4** | 위반 시 코드 품질 점진 저하 | 취향 |

**ROI 판단**: 강제 비용 × 위반 빈도. 비용이 낮고 빈도가 높은 불변식부터.

**원칙**:
- 경계만 강제하고 내부는 자율 — 불변 조건(종속성 방향, 네이밍, 파싱 경계)을 기계적으로 강제하되, 구현 방법은 에이전트에게 위임
- 공유 유틸리티 패키지로 불변식 중앙화 — 커스텀 헬퍼 대신 shared utility를 선호하여 규칙이 한 곳에서 관리됨
- 제약이 중요한 부분과 그렇지 않은 부분을 명확히 구분

## 강제 메커니즘 매핑

SKILL.md의 방법론 우선순위(불가능하게 → 감지 → 수거)와 카테고리의 매핑:

| 카테고리 | Rule (작성 가이드) | Lint/Test (기계적 검증) | Hook (즉시 차단) | 백그라운드 (드리프트 스캔) |
|----------|-------------------|------------------------|-----------------|--------------------------|
| **구조** | rules/architecture.md | ArchUnit, eslint-plugin-boundaries, import-linter | — | 의존 그래프 정기 스캔 |
| **데이터 무결성** | rules/data-boundary.md | 스키마 검증 테스트, migration lint | PreToolUse (migration 파일 보호) | — |
| **API 계약** | rules/api-contract.md | OpenAPI diff (CI), 계약 테스트 | — | API 호환성 정기 검증 |
| **보안** | rules/security.md | SAST, secret scanner, 인증 테스트 | PreToolUse (secret 파일 보호) | 정기 취약점 스캔 |
| **취향** | rules/conventions.md | 커스텀 린트 규칙 (에러 메시지 = 수정 지침) | pre-commit (fmt+lint) | 품질 등급 추적 |
| **운영** | rules/operational.md | 성능 테스트, 트레이싱 검증 | — | SLA 모니터링, 옵저버빌리티 점검 |

**핵심**: 같은 불변식을 Rule(정의) → Lint/Test(강제) → 백그라운드(수거)가 공유해야 드리프트를 방지할 수 있다.

## 안티패턴

| 안티패턴 | 문제 | 대안 |
|----------|------|------|
| 모든 것을 불변식으로 선언 | 과잉 강제 → 유연성 소멸, 개발 속도 급감 | 판단 기준("깨지면 실질적 문제?") 적용. No면 rule로 충분 |
| 구조 불변식만 관리 | 보안, 데이터 무결성 등 다른 유형 방치 | 6개 카테고리로 커버리지 점검 |
| 정의만 하고 강제 없음 | 문서만 있고 CI 없음. 시간이 지나면 위반 누적 | "규칙 부족 → rule로, rule 부족 → 코드로" 승격 |
| 취향을 구조 수준으로 강제 | 린트로 충분한 것을 아키텍처 테스트로 강제하면 과잉 | 취향은 린트, 구조는 아키텍처 테스트로 메커니즘 분리 |
| "왜"가 없는 규칙 | 맥락 없는 규칙은 edge case에서 잘못 적용됨 | 모든 불변식에 Why를 기록. 이유를 알아야 판단 가능 |
