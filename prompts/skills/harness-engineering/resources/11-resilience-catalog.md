# Resilience Catalog — Claude 실패 모드 방어

에이전트(Claude)의 알려진 실행 실패 모드와, 하네스 3계층(Hook → Rule → Skill)에서의 방어 패턴.

> 출처: Reddit "Claude Code 100 Hours vs Codex 20 Hours" (14년차 시니어 엔지니어, 8만줄 Python/TS 프로젝트 실전 비교) + 내부 세션 관찰

---

## 실패 모드 카탈로그

| ID | 모드 | 증상 | 심각도 |
|----|------|------|--------|
| F1 | 지시 무시 | CLAUDE.md/rules 규칙을 안 따름. "매 세션마다 발생" | 높음 |
| F2 | 미완료 작업 | N개 항목 중 일부만 처리 후 완료 보고. "8개 중 6개만 하고 끝냄" | 높음 |
| F3 | 증상 패치 | 근본 원인 대신 증상을 수정. "마감에 쫓기는 엔지니어" 패턴 | 중간 |
| F4 | 테스트 임의 수정 | 실패하는 테스트를 무단 수정하여 통과시킴. "5%가 잘못된 동작을 고정" | 높음 |
| F5 | 코드 비대화 | 신규 파일 생성 회피, 기존 파일에 함수 무분별 추가. 600줄+ 방치 | 중간 |
| F6 | 낮은 사고 깊이 | 충분히 분석하지 않고 바로 행동. "행동 전에 생각하지 않음" | 중간 |

## 방어 패턴 상세

### F1: 지시 무시

**근본 원인**: Rule에 Why가 없으면 Claude가 "불필요하다"고 판단하여 스킵. 컨텍스트가 길어지면 초반 지시를 잊음.

**방어 레이어**:

| 레이어 | 패턴 | 예시 |
|--------|------|------|
| Hook | 핵심 위반을 기계적으로 차단 | `--no-verify` 차단, `git commit --amend` 차단 |
| Rule | **Why + 예시** 필수. Why 없는 규칙은 무시 대상 | `Why: 이전 인시던트에서 prod migration 실패` |
| Skill | 세션 시작 프로토콜로 규칙 로드 강제 | CLAUDE.md "첫 메시지 응답 전 반드시 실행" |
| CLAUDE.md | 목차 역할 유지 (~100줄). 길어지면 읽히지 않음 | Rules로 분리, progressive disclosure |

**점검 질문**: "이 Rule에 Why가 있는가? 없으면 Claude가 왜 따라야 하는지 모른다."

### F2: 미완료 작업

**근본 원인**: Claude는 "대부분 했으니 됐다"고 판단하는 경향. 특히 마이그레이션/리팩토링에서 전수 처리를 하지 않음.

**방어 레이어**:

| 레이어 | 패턴 | 예시 |
|--------|------|------|
| Rule | 마이그레이션/리팩토링 시 **대상 전수를 먼저 열거** 후 작업 | "8개 파일 대상 → TaskList에 8개 등록 → 각각 완료 체크" |
| Skill | Phase별 체크리스트 게이트. 미처리 항목이 남으면 다음 Phase 진행 금지 | design-first의 게이트 리뷰 패턴 |
| Skill | **done criteria**를 검증 가능한 형태로 명시 | "모든 파일에서 old_pattern이 Grep으로 0건" |

**점검 질문**: "전수 대상이 열거되어 있는가? 완료 기준이 기계적으로 검증 가능한가?"

### F3: 증상 패치

**근본 원인**: 빠르게 "해결"하려는 경향. 에러 메시지만 보고 가장 가까운 수정을 적용.

**방어 레이어**:

| 레이어 | 패턴 | 예시 |
|--------|------|------|
| Rule | **이해 → 계획 → 해결** 순서 강제. "단순한 해법부터" | CLAUDE.md 대원칙 |
| Skill | 원인 분석 Phase를 설계에 포함 | refactoring skill의 Characterization Test |
| Rule | 수정 시 **변경 이유(Why)를 커밋 메시지에** 명시 | "Fix: X because Y" 패턴 |

**점검 질문**: "이 수정이 증상인가 원인인가? 왜 이 코드가 이렇게 되어있었는지 이해했는가?"

### F4: 테스트 임의 수정

**근본 원인**: 테스트 실패 → "통과시켜야 한다"는 목표 지향. 테스트가 맞고 코드가 틀릴 가능성을 고려하지 않음.

**방어 레이어**:

| 레이어 | 패턴 | 예시 |
|--------|------|------|
| Hook | test/spec 파일 Edit 시 **nudge 메시지 주입** | "테스트 파일 수정 감지. 테스트 실패의 원인이 코드인지 테스트인지 먼저 판단하세요" |
| Rule | "테스트 실패 시 원인 분석 우선. 테스트 수정은 원인이 테스트에 있을 때만" | `.claude/rules/anti-patterns.md` |
| Rule | "assertion을 삭제/완화하여 테스트를 통과시키지 않는다" | 구체적 금지 |

**점검 질문**: "test 파일에 대한 Edit 보호가 있는가? (Hook 또는 Rule)"

### F5: 코드 비대화

**근본 원인**: 새 파일 생성보다 기존 파일에 추가하는 것이 "안전하다"고 판단. 파일 크기에 대한 피드백이 없음.

**방어 레이어**:

| 레이어 | 패턴 | 예시 |
|--------|------|------|
| Hook | PostToolUse Edit — 파일 라인 수 체크, 임계값 초과 시 경고 | "이 파일이 {N}줄입니다. SRP 위반 가능성 — 분리를 고려하세요" |
| Rule | 파일당 라인 수 가이드라인 명시 | "500줄 초과 시 분리 검토" (프로젝트별 조정) |
| Rule | 새 기능 추가 시 "기존 파일 vs 신규 파일" 결정을 명시적으로 | "단일 책임 원칙: 새 책임 = 새 파일" |

**점검 질문**: "파일 크기 피드백 메커니즘이 있는가? (Hook 권장)"

### F6: 낮은 사고 깊이

**근본 원인**: 토큰 효율 최적화로 "바로 코드 작성"하는 경향. 복잡한 문제에서도 분석 Phase를 건너뜀.

**방어 레이어**:

| 레이어 | 패턴 | 예시 |
|--------|------|------|
| Skill | **게이트 리뷰**: 분석 결과를 사용자에게 보여주고 승인 후 진행 | design-first Phase 0 견적 |
| Skill | **Plan-before-act**: 구현 전 계획 Phase 필수 | plan-review, design-first |
| Rule | "3개 이상이면 TaskCreate로 목록화" | CLAUDE.md 대원칙 |

**점검 질문**: "복잡한 작업에 Plan Phase가 있는가? 사용자 승인 게이트가 있는가?"

## Resilient Skill Template

design-first에서 추출한 "Claude가 단계를 건너뛸 수 없는" 스킬 구조 패턴.
스킬 생성/리팩토링 시 이 패턴을 참조한다.

### 필수 요소

| # | 요소 | 역할 | 예시 (design-first) |
|---|------|------|---------------------|
| 1 | **파일 기반 상태** | Phase 완료마다 파일 저장. compaction 후 복원 가능 | `.claude/designs/{feature}/phase-N-*.md` |
| 2 | **AskUserQuestion 게이트** | Phase 간 명시적 사용자 승인. Claude가 혼자 다음으로 넘어갈 수 없음 | "동의하시나요? 1. 동의 2. Phase N부터 3. 스킵" |
| 3 | **Red Flags** | Phase별 "이 상태면 진행 금지" 조건. 자기 검증 | "기능 목록이 10개 초과 — v1 범위 축소 필요" |
| 4 | **REQUIRED / OPTIONAL 태그** | 어떤 Phase가 스킵 가능한지 명시. 스킵 판단 기준도 포함 | Phase 0 라우팅에서 시작 Phase 결정 |
| 5 | **done criteria** | 각 Phase 완료 기준이 검증 가능한 형태 | "Grep으로 old_pattern 0건 확인" |

### 안티패턴

| 안티패턴 | 문제 | 대안 |
|----------|------|------|
| "아래 단계를 순서대로 실행한다" | Claude가 한번에 압축하여 모든 단계를 한 응답에 처리 | Phase별 AskUserQuestion 게이트로 분리 |
| 게이트 없는 선형 흐름 | 중간 검증 없이 끝까지 달림 | 최소 2개 게이트 (시작 확인 + 완료 전 확인) |
| 사용자에게 보이지 않는 내부 상태 | compaction에서 유실, 세션 재개 시 처음부터 | 파일 기반 상태 + 복원 로직 명시 |
| "MUST"만 있고 Why 없음 | Claude가 "불필요하다"고 판단하여 무시 | Why + Action 형태로 작성 |
| 모든 Phase가 REQUIRED | 단순 작업에서 과도한 오버헤드 | 복잡도 기반 라우팅 (Phase 0에서 시작점 결정) |

### 회복력 등급

| 등급 | 기준 | 예시 |
|------|------|------|
| **High** | 파일 상태 + 게이트 2개+ + Red Flags + done criteria | design-first, qa |
| **Medium** | 체크리스트 있으나 게이트 암시적, 파일 상태 없음 | harness-engineering, plan-review |
| **Low** | 선형 흐름, 게이트 없음, 내부 상태만 | debrief, recall, reflect |

Low 등급 스킬은 반드시 High로 올릴 필요는 없다 — 단순한 스킬은 단순한 구조가 맞다. 
**판단 기준**: "이 스킬에서 Claude가 단계를 건너뛰면 결과물 품질에 심각한 영향이 있는가?"

## Audit 점검 기준

Resilience 차원 (0-3점):

| 점수 | 기준 |
|------|------|
| 3 | F1-F6 중 4개+ Hook/Rule 방어. 프로젝트 스킬이 Resilient Template 따름 |
| 2 | F1-F6 중 2-3개 방어. 일부 스킬만 게이트 있음 |
| 1 | Rule만 존재하고 Hook 없음. 스킬에 게이트 없음 |
| 0 | 실패 모드 방어 없음 |

## Setup 가이드

프로젝트에 Resilience 방어를 설치하는 순서:

1. **Rule 먼저** — `.claude/rules/anti-patterns.md` 생성 (비용 제로, 즉시 효과)
   - 프로젝트에서 가장 빈번한 실패 모드 2-3개 선택
   - 각 규칙에 Why + Action 포함
2. **Hook 추가** — 기계적 강제가 필요한 것만
   - F5(코드 비대화): PostEdit 파일 크기 경고 — 거의 모든 프로젝트에 유용
   - F4(테스트 보호): PreEdit test 파일 nudge — 테스트가 있는 프로젝트
3. **Skill 점검** — 기존 프로젝트 스킬의 회복력 등급 확인
   - High 영향 스킬(설계, QA)이 Low 등급이면 리팩토링 제안
