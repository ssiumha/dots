# plan-review

실행 전 계획을 코드베이스 기준으로 비판적 검증하여 PASS/WARN/FAIL 리포트를 생성한다.

## Quick Reference

```
/plan-review              # .claude/plans/ 최신 파일 리뷰
/plan-review {file}       # 지정 파일 리뷰
/plan-review --quick      # Check 1-3만 수행, Check 4-6 및 Domain Review 스킵
```

## 핵심 철학

1. **실행 전 검증**: 코드 한 줄 쓰기 전에 계획의 결함을 찾는다
2. **Codebase-grounded**: 추측이 아닌 실제 파일/패턴 확인으로 검증한다
3. **분석만, 수정 안 함**: 리포트와 제안만 제공하고 계획을 직접 수정하지 않는다
4. **Chesterton's Fence**: 기존 코드의 존재 이유를 먼저 이해한 뒤 변경을 평가한다

---

## Phase 1: 계획 수집

### 1-1. 계획 파일 결정

인자가 있으면 해당 파일을, 없으면 아래 순서로 탐색한다:

1. `.claude/plans/` 디렉토리에서 가장 최근 수정된 `.md` 또는 `.yaml` 파일
2. 현재 대화의 plan mode에서 확정된 계획 (대화 컨텍스트에서 직접 참조)
3. 못 찾으면 사용자에게 경로를 물어본다

### 1-2. 계획 파싱

계획 포맷(YAML/Markdown)에 따라 아래와 같이 요소를 추출한다:

| 요소 | YAML 필드 | Markdown 헤더/패턴 |
|------|-----------|-------------------|
| **목표** | `goal` | `## 목표` / `## Goal` / Summary 섹션 |
| **태스크 목록** | `tasks[]` | `## Tasks` 테이블 또는 체크리스트 |
| **참조 파일** | 태스크 내용에서 추출 | 태스크 내용에서 추출 |
| **검증 방법** | `tasks[].verify` | Verify 컬럼 또는 검증 섹션 |
| **사용자 원본 요청** | `context` / `request` | 계획 상단 Context 섹션 (있으면) |

## Phase 2: 6-Point Review

각 체크를 순서대로 수행한다. `--quick` 모드에서는 Check 1, 2, 3만 수행한다.

> 각 체크의 상세 PASS/WARN/FAIL 기준은 `resources/01-review-checklist.md`를 참조한다.

### Check 1: Scope Alignment (범위 정합성)

사용자 요청 대비 계획의 범위가 적절한지 검증한다.

- [ ] 계획의 목표가 사용자 요청을 충족하는가?
- [ ] 요청에 없는 불필요한 작업이 포함되어 있지 않은가? (scope creep)
- [ ] 요청 대비 과도한 설계가 아닌가? (over-engineering: 불필요한 추상화, YAGNI 위반)
- [ ] 요청된 내용 중 누락된 항목이 없는가?
- [ ] 목표가 구체적이고 측정 가능한가?

**검증 방법**: 계획 텍스트와 사용자 요청을 대조 분석

### Check 2: Codebase Validation (코드베이스 정합성)

계획이 참조하는 파일, 패턴, API가 실제로 존재하는지 확인한다.

- [ ] 참조된 파일 경로가 실제로 존재하는가? → `Glob`
- [ ] 참조된 함수/클래스/변수가 존재하는가? → `Grep`
- [ ] import 경로, 모듈 구조가 현재 코드와 일치하는가?
- [ ] 사용하려는 프레임워크/라이브러리 버전이 맞는가?

**검증 방법**: Glob, Grep, Read로 실제 코드베이스 확인

### Check 3: Risk / Impact (위험도 / 영향 범위)

기존 기능에 미치는 영향과 되돌리기 난이도를 평가한다.

- [ ] 기존 기능을 깨뜨릴 가능성이 있는가?
- [ ] 되돌리기 어려운 작업(DB 마이그레이션, 파일 삭제 등)이 포함되어 있는가?
- [ ] dirty state(커밋되지 않은 변경)와 충돌 가능성이 있는가? → `git status`
- [ ] 외부 시스템(API, 서비스)에 영향을 주는가?
- [ ] 수정 파일 수 대비 테스트 커버리지가 충분한가?

**검증 방법**: git status, Grep으로 참조 관계 추적, 영향 범위 분석

> Note: 기존 코드 변경의 타당성(왜 이 코드가 있는가)은 Check 5에서 심화 분석한다. 여기서는 영향 범위와 되돌리기 난이도에 집중한다.

### Check 4: Completeness (완성도)

계획이 빠뜨린 에러 핸들링, 엣지 케이스, 후속 작업을 점검한다.

> Note: Check 3에서는 수정 규모 대비 테스트 존재 여부(위험 완화)를 보고, 여기서는 테스트 계획의 구체성과 충분성(완성도)을 본다.

- [ ] 에러/예외 처리가 계획에 포함되어 있는가?
- [ ] 엣지 케이스(빈 입력, 대량 데이터, 권한 등)를 고려했는가?
- [ ] 필요한 테스트(단위/통합)가 계획에 포함되어 있는가?
- [ ] 의존성 설치, 설정 변경 등 환경 준비가 포함되어 있는가?
- [ ] 태스크 간 의존성이 올바른가? (순서 오류, 누락된 의존성)

**검증 방법**: 계획 텍스트 분석, 참조 파일의 에러 패턴 확인

### Check 5: Chesterton's Fence (기존 패턴 존중)

기존 코드를 변경하거나 제거하는 경우, 그 이유를 이해했는지 확인한다.

- [ ] 삭제/대체하려는 코드의 원래 목적을 파악했는가?
- [ ] 기존 패턴과 다른 방식을 선택한 이유가 명시되어 있는가?
- [ ] 프로젝트 컨벤션(네이밍, 디렉토리 구조, 코드 스타일)을 따르는가?
- [ ] 과거 결정(ADR, 커밋 메시지, 주석)과 충돌하지 않는가?

**검증 방법**: Read로 기존 코드 패턴 확인, git log로 이력 추적

### Check 6: Verification Plan (검증 계획)

계획이 "완료"를 어떻게 확인할지 구체적으로 명시했는지 점검한다.

- [ ] 각 태스크별 검증 방법이 명시되어 있는가?
- [ ] 자동 검증(테스트, lint)과 수동 검증이 구분되어 있는가?
- [ ] 검증 명령어가 구체적인가? (예: `npm test` vs "테스트 실행")
- [ ] 통합 검증(전체 동작 확인) 단계가 있는가?

**검증 방법**: 계획 텍스트에서 검증 관련 항목 분석

## Phase 2.5: Domain Review

계획 내용을 분석하여 보안/설계/UX 분야별 전문 리뷰를 수행한다. `--quick` 모드에서는 이 Phase를 건너뛴다.

### 2.5-0. 관련성 판단 (스킵 로직)

각 분야의 관련성을 판단하여 **관련 있는 분야만 실행**한다. 모든 분야가 스킵되면 Phase 3으로 바로 진행한다.

| 분야 | 트리거 조건 (1개 이상 해당 시 실행) | 스킵 예시 |
|------|-------------------------------------|-----------|
| Security | 인증/인가 변경, API 엔드포인트 추가/수정, 외부 데이터 수신, 패키지 추가, 환경변수/시크릿 관련, DB 스키마 변경 | 내부 유틸 리팩토링, 문서 수정, 순수 UI 스타일링 |
| Design | 2개+ 컴포넌트 변경, 새 모듈/서비스 생성, 의존성 구조 변경, 인터페이스/타입 변경 | 단일 파일 버그 수정, 설정값 변경, 텍스트 수정 |
| Architecture | 레이어 경계를 넘는 변경, 새 모듈/패키지 도입, 의존성 방향 변경, 도메인 모델 변경, 외부 시스템 연동 추가 | 단일 모듈 내부 수정, 스타일링, 설정값 변경 |
| UX | 사용자 인터페이스 변경, 에러 메시지 수정, 새 화면/흐름 추가, 성능 관련 변경, CLI 인터페이스 변경 | 백엔드 내부 로직, 인프라 변경, 테스트 코드만 수정 |
| Test | 새 기능 추가, 기존 로직 변경, API 변경, 리팩토링, 테스트 전략 변경 | 문서 수정, 설정값 변경, 순수 UI 스타일링 |

### 2.5-1. 병렬 Agent 호출

관련 분야의 Agent만 **단일 메시지에서 병렬 호출**한다 (1~5개).

- `subagent_type`: `Explore` (read-only, Glob/Grep/Read 가능)
- 각 Agent 프롬프트에 포함: 계획 전문 + 사용자 원본 요청 + 해당 분야 체크리스트 + 출력 포맷

#### Agent 1: Security Reviewer (보안)

평가 항목:
- 인증/인가: 변경이 인증 흐름에 영향을 주는가? 권한 검증이 누락되지 않았는가?
- 입력 검증: 사용자 입력/외부 데이터에 대한 검증이 계획에 포함되어 있는가?
- 데이터 노출: 민감 데이터(비밀번호, 토큰, PII)가 로그/응답에 노출될 위험이 있는가?
- 의존성 보안: 새 패키지/라이브러리 추가 시 알려진 취약점이 있는가?
- API 보안: CORS, rate limiting, 인증 미들웨어 적용이 필요한가?
- 시크릿 관리: 환경변수/설정 파일에 하드코딩된 시크릿이 없는가?

#### Agent 2: Design Reviewer (설계)

평가 항목:
- 책임 분리: 각 컴포넌트가 단일 책임을 가지는가? (SRP)
- 결합도: 컴포넌트 간 결합이 느슨한가? 순환 의존이 없는가?
- 기존 패턴 일관성: 프로젝트의 기존 아키텍처 패턴을 따르는가?
- 확장성: 향후 변경 시 수정 범위가 최소화되는 구조인가? (OCP)
- 복잡도: 과설계(YAGNI) 또는 과소설계가 아닌가?
- 테스트 용이성: 설계가 단위 테스트를 쉽게 작성할 수 있는 구조인가?

#### Agent 3: Architecture Reviewer (아키텍처)

평가 항목:
- 레이어 경계: 변경이 레이어 경계(presentation/domain/infra)를 존중하는가? 레이어를 넘는 직접 참조가 없는가?
- 의존성 방향: 모듈 간 의존성이 안쪽(도메인)→바깥(인프라) 방향을 위반하지 않는가?
- 도메인 격리: 비즈니스 로직이 프레임워크/인프라 세부사항에 직접 의존하지 않는가?
- 모듈 응집도: 새 모듈/패키지가 명확한 경계와 단일 책임을 가지는가?
- 시스템 통합: 외부 시스템 연동 시 Anti-corruption Layer나 어댑터 패턴을 적용했는가?
- 확장성: 향후 요구사항 변경 시 수정 범위가 특정 레이어/모듈에 격리되는가?

#### Agent 4: UX Reviewer (사용자 경험)

평가 항목:
- 사용자 흐름: 변경이 기존 사용자 흐름을 방해하지 않는가?
- 에러 피드백: 에러 발생 시 사용자에게 명확한 메시지/안내를 제공하는가?
- 성능 체감: 응답 시간, 로딩 상태 등 체감 성능에 영향이 있는가?
- 일관성: 기존 UI/UX 패턴(네이밍, 흐름, 레이아웃)과 일관되는가?
- 접근성: 키보드 네비게이션, 스크린 리더 등 접근성을 고려했는가?
- 엣지 케이스: 빈 상태, 대량 데이터, 오프라인 등 엣지 상황에서의 UX를 고려했는가?

#### Agent 5: Test Reviewer (테스트)

평가 항목:
- 테스트 종류 적절성: 변경 규모에 맞는 테스트 레벨(unit/integration/e2e)이 계획에 포함되었는가?
- 커버리지 범위: 핵심 경로(happy path)와 경계 조건(boundary)에 대한 테스트가 계획되었는가?
- 테스트 격리: 테스트 간 의존성이 없는가? 외부 의존(DB, API, 파일시스템) 격리 전략이 적절한가?
- 기존 테스트 영향: 계획의 변경으로 기존 테스트가 깨질 가능성이 있는가? 수정이 필요한 테스트가 식별되었는가? → `Grep`으로 기존 테스트 파일에서 변경 대상 함수/클래스 참조 확인
- TDD 적합성: 테스트 선행 작성이 가능한 태스크를 식별하고, TDD 진행을 제안
- Mock 남용: 과도한 mock으로 실제 동작과 괴리될 위험이 있는가? 통합 테스트로 대체 가능한 mock이 있는가?

### 2.5-2. Agent 출력 포맷

각 Agent는 아래 포맷으로 결과를 반환한다:

```
## [분야] Review
**Result**: PASS | WARN | FAIL
**Summary**: 한줄 요약

### 상세 (WARN/FAIL만 작성)
**문제**: 구체적 문제 설명
**영향**: 이 문제가 실행에 미치는 영향
**제안**: 수정 방향
```

## Phase 3: 리포트 생성

### 3-0. 데이터 통합

Phase 2 (6-Point Review) 결과와 Phase 2.5 (Domain Review) Agent 출력을 아래 규칙으로 통합한다:

1. 각 Agent의 `Result` → 분야별 리뷰 테이블의 `Result` 컬럼
2. 각 Agent의 `Summary` → 분야별 리뷰 테이블의 `Summary` 컬럼
3. WARN/FAIL인 Agent의 `상세` 블록 → Phase 3-3 FAIL/WARN 상세에 포함 (기본 체크 항목과 동일 포맷)
4. 스킵된 분야 → `N/A — 해당 없음 (스킵 사유)`

### 3-1. 요약 테이블

```markdown
## Plan Review Report

### 기본 체크
| # | Check | Result | Summary |
|---|-------|--------|---------|
| 1 | Scope Alignment | PASS/WARN/FAIL | 한줄 요약 |
| 2 | Codebase Validation | PASS/WARN/FAIL | 한줄 요약 |
| 3 | Risk / Impact | PASS/WARN/FAIL | 한줄 요약 |
| 4 | Completeness | PASS/WARN/FAIL | 한줄 요약 |
| 5 | Chesterton's Fence | PASS/WARN/FAIL | 한줄 요약 |
| 6 | Verification Plan | PASS/WARN/FAIL | 한줄 요약 |

### 분야별 리뷰
| 분야 | Result | Summary |
|------|--------|---------|
| Security | PASS/WARN/FAIL/N/A | 한줄 요약 또는 "해당 없음" |
| Design | PASS/WARN/FAIL/N/A | 한줄 요약 또는 "해당 없음" |
| Architecture | PASS/WARN/FAIL/N/A | 한줄 요약 또는 "해당 없음" |
| UX | PASS/WARN/FAIL/N/A | 한줄 요약 또는 "해당 없음" |
| Test | PASS/WARN/FAIL/N/A | 한줄 요약 또는 "해당 없음" |
```

스킵된 분야는 `N/A — 해당 없음 (사유)` 형태로 표기한다. `--quick` 모드에서는 분야별 리뷰 테이블 자체를 생략한다.

### 3-2. 구조 관계도

계획의 변경 범위를 시각적으로 표현하는 ASCII 관계도를 생성한다.

#### 생성 기준

아래 우선순위로 판단하여, 해당하는 **첫 번째** 유형을 생성한다. 복수 유형이 동등하면 가장 복잡한 것 하나만 선택한다.

| 우선순위 | 조건 | 관계도 유형 |
|----------|------|------------|
| 1 | API/데이터 흐름 변경 | 데이터 플로우 다이어그램 (`Client -> API -> DB`) |
| 2 | 서로 호출/의존하는 모듈 2개+ 변경 | 컴포넌트 의존 관계도 ("모듈" = 하나의 책임 단위: 서비스, 컨트롤러, 리포지토리 등) |
| 3 | 새 모듈/파일 추가 | 파일/디렉토리 구조도 (트리 형태 + `(+)` 표기) |
| — | 단일 파일 수정, 설정값 변경 | 생략 |

#### 표기 규칙

- `[컴포넌트]` : 기존 컴포넌트
- `[컴포넌트] (+)` : 신규 추가
- `[컴포넌트] (*)` : 수정 대상
- `-->` : 의존/호출 방향
- `- - ->` : 간접 의존 (이벤트, 메시지 등)

#### 예시

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ AuthRouter  │────>│ AuthCtrl (*) │────>│ AuthService │
│    (+)      │     └──────────────┘     │    (+)      │
└─────────────┘            │             └─────────────┘
                           │                    │
                           v                    v
                    ┌──────────────┐     ┌─────────────┐
                    │  Middleware  │     │  UserRepo   │
                    │    (*)      │     └─────────────┘
                    └──────────────┘
```

단일 파일 수정, 설정값 변경 등 관계도가 불필요한 경우 이 섹션을 생략한다.

### 3-3. FAIL/WARN 상세

FAIL 또는 WARN인 항목만 상세 설명을 작성한다 (기본 체크 + 분야별 리뷰 모두 포함):

```markdown
### FAIL: Check 2 — Codebase Validation

**문제**: `src/utils/helper.ts` 파일이 존재하지 않음
**영향**: Task 3에서 이 파일을 import하므로 빌드 실패
**제안**: 실제 경로 `src/lib/helpers.ts`로 수정 필요
```

```markdown
### WARN: Security Review

**문제**: API 엔드포인트에 인증 미들웨어 적용 계획 누락
**영향**: 미인증 사용자가 보호 자원에 접근 가능
**제안**: 라우터에 authMiddleware 추가 필요
```

### 3-4. 개선 제안

FAIL/WARN 항목에 대해 구체적인 수정 방향을 제안한다. 직접 수정하지 않는다.

### 3-5. 총평

실행된 항목만 합산한다. 스킵/N/A 항목은 PASS/WARN/FAIL 카운트에서 제외한다.

- **일반 모드**: Check 1-6 + 실행된 Domain Review. 3개 Domain 모두 N/A이면 Check 1-6만으로 판정.
- **Quick 모드**: Check 1-3만으로 판정. Domain Review는 실행하지 않으므로 합산 대상 아님.

| 조건 | 총평 |
|------|------|
| 실행 항목 중 FAIL 0개, WARN 0-1개 | **PASS** — 실행 권장 |
| 실행 항목 중 FAIL 0개, WARN 2개 이상 | **WARN** — 수정 권장 후 실행 |
| 실행 항목 중 FAIL 1개 이상 | **FAIL** — 수정 필수 |

### 3-6. 병렬화 분석

Phase 3 리포트의 마지막 섹션으로 출력한다. `--quick` 모드에서는 생략한다.

#### 3-6-0. 스킵 기준

| 조건 | 처리 |
|------|------|
| 태스크 3개 미만 | 스킵 — "단독 실행 권장" |
| 총평 FAIL | 스킵 — 계획 수정이 우선 |
| 모든 태스크가 직렬 의존 | "병렬화 불가 — 전체가 순차 의존 체인" 표기 |

#### 3-6-1. 의존성 그래프 분석

계획의 태스크를 의존성 기준으로 **병렬 트랙**을 식별한다:

1. 태스크 간 의존성(blockedBy)에서 DAG 구성
2. 동시 실행 가능한 독립 태스크 그룹(병렬 트랙) 식별
3. 크리티컬 패스 길이 계산

```
예시:
Task 1: DB 스키마 추가           ─┐
Task 2: API 엔드포인트 구현       ─┤─ Track A (backend)
Task 3: 인증 미들웨어 추가        ─┘  (1→2→3 순차)

Task 4: UI 컴포넌트 구현          ─┐
Task 5: 폼 검증 추가              ─┘─ Track B (frontend)
                                      (4→5 순차)

Task 6: E2E 테스트 작성           ── Track C (blocked by A+B)

→ 병렬 트랙: A, B (동시 실행 가능)
→ 크리티컬 패스: 3단계 (Track A)
→ 병렬화 이득: 3+2+1 순차 = 6단계 → 병렬 시 3+1 = 4단계
```

#### 3-6-2. 파일 오너십 분석

각 병렬 트랙이 수정하는 파일 목록을 추출하고 **충돌 여부**를 검사한다:

| 상태 | 판정 |
|------|------|
| 트랙 간 수정 파일 겹침 없음 | SAFE — 병렬 실행 가능 |
| 공유 파일 있으나 수정 범위가 다른 함수/섹션 | CAUTION — worktree 격리 권장 |
| 동일 함수/영역 수정 | CONFLICT — 순차 실행 또는 태스크 재분할 필요 |

#### 3-6-3. 실행 전략 제안

분석 결과에 따라 3가지 중 하나를 제안한다:

| 조건 | 전략 | 설명 |
|------|------|------|
| 병렬 트랙 1개 | **Solo** | 단독 실행. 팀 불필요 |
| 병렬 트랙 2개+, 파일 충돌 없음 | **Team** | 트랙당 1 teammate 배정 |
| 병렬 트랙 2개+, 파일 충돌 있음 | **Team + Worktree** | teammate별 worktree 격리 |

#### 3-6-4. 리포트 출력 포맷

```markdown
### 병렬화 분석

**실행 전략**: Team (2 teammates 권장)
**병렬화 이득**: 6단계 → 4단계 (33% 단축)

| Track | Teammate | Tasks | 수정 파일 | 충돌 |
|-------|----------|-------|-----------|------|
| A: Backend | backend-dev | #1→#2→#3 | src/api/, src/db/ | — |
| B: Frontend | frontend-dev | #4→#5 | src/ui/, src/components/ | — |
| C: Test | (Track A,B 완료 후) | #6 | tests/ | — |

#### 스폰 가이드

**Teammate 1: backend-dev**
- model: sonnet
- 담당: Task #1, #2, #3 (순차)
- 수정 범위: `src/api/`, `src/db/`
- 완료 기준: API 테스트 통과

**Teammate 2: frontend-dev**
- model: sonnet
- 담당: Task #4, #5 (순차)
- 수정 범위: `src/ui/`, `src/components/`
- 완료 기준: 컴포넌트 렌더링 확인
```

## Phase 4: 후속 안내

총평 결과와 병렬화 분석 결과에 따라 다음 행동을 안내한다:

| 총평 | 병렬화 분석 결과 | 안내 |
|------|-----------------|------|
| **PASS** | Solo | "계획에 큰 문제가 없습니다. 실행을 진행하세요." |
| **PASS** | Team | "계획에 큰 문제가 없습니다. 위 스폰 가이드로 팀을 구성하면 {N}% 단축 가능합니다." |
| **WARN** | Solo | "아래 항목을 확인한 뒤 실행하세요: [WARN 목록]. `/plan-creator`로 계획을 수정할 수 있습니다." |
| **WARN** | Team | "아래 항목을 확인한 뒤, 위 스폰 가이드로 팀을 구성하세요." |
| **FAIL** | (무시) | "수정이 필요합니다. 병렬화는 계획 수정 후 재분석합니다. `/plan-creator`로 계획을 수정한 뒤 다시 `/plan-review`를 실행하세요." |

보안 관련 WARN/FAIL이 있으면 `/review-security` 실행을 추가 권장한다.
(기준: 인증/인가, 입력 검증, 암호화, API 노출, 환경 변수/시크릿 관련 변경이 계획에 포함된 경우)

User: /plan-review

User: /plan-review --quick

## 연동

```
prompt-review (요청 정제)
  → plan-creator (계획 생성)
    → plan-review (계획 검증)  ← 현재 skill
      → 실행
        → reflect (사후 회고)
```

| Skill | 관계 | 설명 |
|-------|------|------|
| `plan-creator` | 보완 | 계획 생성 후 검증. FAIL 시 plan-creator로 수정 |
| `reflect` | 사전↔사후 | plan-review는 실행 전, reflect는 실행 후 |
| `review-security` | 위임 | 보안 관련 심층 검토는 review-security에 위임 |
| `prompt-review` | 선행 | 요청이 모호하면 prompt-review로 먼저 정제 |
