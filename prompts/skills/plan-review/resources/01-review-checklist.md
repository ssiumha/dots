# Plan Review Checklist — PASS/WARN/FAIL 기준

각 체크 항목별 구체적 판정 기준과 예시.

---

## Check 1: Scope Alignment (범위 정합성)

### PASS

- 계획의 목표가 사용자 요청과 1:1 매핑됨
- 불필요한 추가 작업 없음
- 복잡도가 요청 규모에 비례
- 누락된 요구사항 없음

> 예: 사용자가 "로그인 API 추가"를 요청, 계획에 로그인 엔드포인트 + 인증 미들웨어 + 테스트가 포함. 직접적인 구현 위주.

### WARN

- 사소한 추가 작업이 포함됨 (리팩토링, 주석 추가 등)
- 목표가 약간 추상적이나 방향은 맞음
- 약간의 과도 설계 (당장 불필요한 인터페이스/추상화 포함이나, 구현 비용이 낮음)

> 예: "로그인 API 추가" 요청에 "기존 라우터 구조 개선"이 포함됨. 또는 단순 함수면 충분한데 Strategy 패턴을 도입하나 코드량 차이가 미미.

### FAIL

- 요청과 무관한 작업이 계획의 상당 부분을 차지 (가이드라인: 태스크 1/3 이상이 무관한 작업. 단, 5개 이하 소규모 계획에서는 비율보다 핵심 태스크 누락 여부를 우선 판단)
- 요청 대비 과도한 설계: 불필요한 추상 레이어, DI 컨테이너, 팩토리 패턴 등 YAGNI 위반
- 핵심 요구사항이 누락됨
- 목표가 모호하여 완료 기준을 판단할 수 없음

> 예: "설정 파일 읽기" 요청에 플러그인 시스템 + 추상 팩토리 + 이벤트 버스 설계가 포함. 또는 "로그인 API 추가" 요청에 "전체 인증 시스템 재설계"가 포함.

## Check 2: Codebase Validation (코드베이스 정합성)

### PASS

- 참조된 모든 파일 경로가 실제 존재
- 함수/클래스/변수명이 현재 코드와 일치
- import 경로, 모듈 구조가 정확

> 예: `src/routes/auth.ts`를 수정 대상으로 명시, Glob으로 확인 시 실제 존재

### WARN

- 1-2개 파일 경로가 부정확하나 의도는 파악 가능
- 약간 변경된 API 시그니처 (파라미터 순서 등)

> 예: `src/utils/hash.ts`로 참조했으나 실제는 `src/lib/hash.ts` — 의도는 명확

### FAIL

- 존재하지 않는 파일을 핵심 의존성으로 참조
- 현재 코드에 없는 함수/클래스를 호출
- 프레임워크 버전이 달라 API가 다름

> 예: `UserService.findById()` 호출을 계획하나, 실제 코드에는 `UserRepository.get()`만 존재

---

## Check 3: Risk / Impact (위험도 / 영향 범위)

### PASS

- 영향 범위가 명확하고 제한적
- 되돌리기 쉬운 변경만 포함
- dirty state 충돌 없음

> 예: 새 파일 2개 추가, 기존 파일 1개에 import 추가. git status clean.

### WARN

- 다수 파일(5개+) 수정이지만 테스트 커버리지가 충분
- dirty state와 수정 대상 파일이 겹침
- 외부 API 호출 변경이 포함되나 하위 호환

> 예: 10개 파일 수정이나 모두 테스트 존재. git status에 수정 대상 1개가 unstaged.

### FAIL

- 되돌리기 어려운 작업 (DB 스키마 변경, 파일 삭제)이 롤백 계획 없이 포함
- 핵심 모듈을 대규모 변경하나 테스트가 없음
- dirty state의 핵심 파일과 수정 대상이 충돌

> 예: DB 마이그레이션 포함이나 down 마이그레이션 미작성. 커밋되지 않은 변경이 있는 `auth.ts`를 대규모 수정 예정.

---

## Check 4: Completeness (완성도)

### PASS

- 에러 핸들링이 계획에 포함
- 엣지 케이스 고려 명시
- 테스트 작성이 태스크에 포함
- 태스크 간 의존성이 올바름

> 예: API 엔드포인트 추가 시 입력 유효성 검사, 에러 응답, 단위 테스트, 통합 테스트가 모두 태스크에 포함

### WARN

- 에러 핸들링이 암시적 (별도 태스크 없이 구현에 포함될 것으로 가정)
- 테스트가 포함되어 있으나 범위가 불충분
- 환경 설정 변경이 언급만 되고 구체적이지 않음

> 예: "적절한 에러 핸들링 추가"라고만 기술, 구체적 에러 시나리오 미명시

### FAIL

- 에러 핸들링 전혀 없음
- 테스트 계획 전혀 없음
- 태스크 의존성에 순환 참조 또는 누락된 선행 작업 존재
- 필수 환경 준비 (패키지 설치, 설정 파일 등) 누락

> 예: 새 패키지를 사용하나 설치가 어떤 태스크에도 없음. Task 3이 Task 2에 의존하나 blockedBy 미설정.

---

## Check 5: Chesterton's Fence (기존 패턴 존중)

### PASS

- 기존 코드 변경 이유가 명시됨
- 프로젝트 컨벤션을 따름
- 과거 결정(ADR, 주석)과 충돌 없음

> 예: "현재 동기 처리 → 비동기로 변경. 이유: 대량 데이터 처리 시 블로킹 발생(#123)"

### WARN

- 기존 패턴과 다른 접근이나 이유가 암시적
- 컨벤션을 대부분 따르나 일부 예외 (네이밍 등)

> 예: 기존 모듈은 class 기반이나 새 모듈만 함수형으로 작성 예정 — 이유 미명시

### FAIL

- 기존 코드를 이유 없이 대체
- 프로젝트 컨벤션을 무시 (다른 디렉토리 구조, 네이밍 등)
- 과거 결정과 직접 충돌하는 변경

> 예: 모든 API 핸들러가 `controllers/` 디렉토리에 있으나 새 핸들러를 `handlers/`에 생성. 기존 `// DO NOT CHANGE: legacy compatibility` 주석이 있는 코드를 수정.

---

## Check 6: Verification Plan (검증 계획)

### PASS

- 각 태스크별 검증 방법이 구체적
- 자동 검증 명령어가 명시됨 (예: `pytest tests/test_auth.py`)
- 통합 검증 단계가 있음

> 예: "Task 2 완료 후 `npm test -- --grep auth` 실행, Task 5 완료 후 전체 `npm test` + 수동 로그인 테스트"

### WARN

- 검증 방법이 있으나 추상적 (예: "테스트 실행")
- 자동/수동 구분이 없음
- 통합 검증 단계 없음 (개별 태스크 검증만)

> 예: "각 단계 완료 후 테스트 확인" — 어떤 테스트를 어떻게 실행하는지 불명확

### FAIL

- 검증 방법이 전혀 없음
- "동작 확인"만 명시되고 구체적 방법 없음
- 테스트가 없는 코드의 검증 계획 없음

> 예: 5개 태스크가 있으나 검증 관련 언급이 전혀 없음

## Domain Review: Architecture (아키텍처)

### PASS

- 레이어 경계가 명확하게 유지됨 (presentation/domain/infra 간 직접 참조 없음)
- 의존성 방향이 올바름 (바깥→안쪽, 인프라→도메인 방향 의존 없음)
- 새 모듈이 기존 모듈 경계와 일관된 구조를 가짐
- 외부 시스템 연동 시 어댑터/ACL 패턴 적용

> 예: 새 결제 서비스 추가 시 `domain/payment/` 인터페이스 정의 + `infra/payment/` 구현체 분리. 의존성이 infra → domain 방향.

### WARN

- 레이어 경계를 넘는 참조가 있으나 범위가 제한적 (1-2곳)
- 모듈 경계가 다소 모호하나 기능적으로 분리됨
- 외부 시스템 직접 호출이 있으나 격리 비용 대비 규모가 작음

> 예: 컨트롤러에서 리포지토리를 직접 호출하는 코드 1곳. 소규모 변경이라 서비스 레이어 추가가 과도할 수 있음.

### FAIL

- 레이어 경계 위반이 구조적 (도메인이 인프라에 직접 의존)
- 순환 의존이 모듈 간 발생
- 비즈니스 로직이 프레임워크/인프라 세부사항에 강결합
- 외부 시스템 변경 시 도메인 코드 수정이 필요한 구조

> 예: 도메인 엔티티가 ORM 어노테이션에 직접 의존. 결제 게이트웨이 변경 시 도메인 로직까지 수정 필요.

## Domain Review: Test (테스트)

### PASS

- 변경 규모에 맞는 테스트 레벨(unit/integration/e2e)이 계획에 포함
- 핵심 경로와 경계 조건에 대한 테스트가 명시
- 기존 테스트 영향이 분석되어 수정 대상이 식별됨
- 테스트 격리 전략이 적절 (외부 의존 mock/stub 명시)
- TDD 적합 태스크가 식별되어 테스트 선행 작성 순서 반영

> 예: API 엔드포인트 추가 시 단위 테스트(서비스 로직) + 통합 테스트(API 호출) 포함. 기존 `test_user_service.py`에서 변경 대상 함수 참조 2건 수정 필요 명시. 서비스 로직은 TDD로 진행 제안.

### WARN

- 테스트 포함되어 있으나 레벨이 부적절 (모두 unit만, 또는 모두 e2e만)
- 기존 테스트 영향 분석 누락 (변경 대상을 참조하는 테스트 미확인)
- Mock 사용이 과도하여 실제 동작과 괴리 위험
- TDD 적합 태스크가 있으나 식별되지 않음

> 예: DB 스키마 변경이 포함되나 기존 통합 테스트 수정 계획 없음. 외부 API 호출을 모두 mock하여 실제 응답 형식 검증 불가.

### FAIL

- 테스트 계획 전혀 없음 (기능 변경에도 테스트 태스크 미포함)
- 기존 테스트가 확실히 깨질 변경인데 수정 계획 없음
- 테스트 격리 전략 없이 외부 의존(DB, API) 직접 호출
- 핵심 비즈니스 로직 변경에 경계 조건 테스트 전무

> 예: 인증 로직 변경인데 테스트 태스크가 없음. 기존 `test_auth.py`가 변경 대상 함수를 10곳에서 참조하나 언급 없음.

## Quick 모드 선정 이유

Quick 모드에서 Check 1, 2, 3만 수행하는 이유:

1. **Check 1 (Scope)**: 방향이 틀리면 나머지가 의미 없음 — 가장 치명적
2. **Check 2 (Codebase)**: 존재하지 않는 파일/함수 기반 계획은 실행 불가 — 즉시 실패
3. **Check 3 (Risk)**: 되돌리기 어려운 작업은 사전 차단 필수 — 피해 최대

Check 4-6은 실행 품질에 영향을 주지만 "실행 불가" 수준의 문제는 아니므로 quick에서 생략한다.

Domain Review (Security/Design/Architecture/UX/Test) 역시 quick에서 생략한다:

4. **Domain Review**: 계획의 품질 심화 검토에 해당 — Quick의 "실행 가능 여부 빠른 판단" 목적과 맞지 않음
