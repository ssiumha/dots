# AMDAHLS-LAW — Amdahl's Law

## 정의

> "The overall performance improvement gained by optimizing a single part of a system is limited by the fraction of time that the improved part is actually used."

처리 시간의 비율이 병렬 가능 `p`, 순차 `1-p`일 때, 코어 `n`개로 얻는 최대 속도 향상:

```
S(n) = 1 / ((1 - p) + p/n)
```

`p=0.9` (병렬 90%)여도 `n → ∞`일 때 한계는 **10배**. 순차 구간이 10%만 있어도 영원히 10배를 못 넘는다. **병렬화의 이득은 빠르게 포화한다.** 같은 논리가 분산 시스템, 비동기 I/O, 파이프라인 최적화, 캐시 병목 제거 모두에 적용된다.

## 핵심 판단

- **"이 워크로드에서 진짜 순차적인 구간이 몇 %인가?"** — 측정 전엔 모름
- **"병렬화로 얻을 이론적 최대 속도 향상은 얼마인가?"** — 계산하면 실망할 때 많음
- **"병목이 병렬 구간인가, 순차 구간인가?"** — 병렬 구간만 최적화하면 한계가 빨리 옴
- **"병렬화 비용(조율, 동기화, 재시도)이 이득을 상쇄하는가?"** — 오버헤드 포함해 재계산

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 병렬 블록 내부에 공유 락이 있음 | `grep -r "Mutex\|Lock\|synchronized"` 병렬 경로에서 |
| `Promise.all`/`gather` 안에 순차 `await`가 섞임 | 비동기 흐름 그래프 — await 전후 관계 |
| 워커 N개지만 단일 큐/DB에 수렴 | 팬아웃 이후의 수렴점 확인 |
| 배치 처리에서 전역 정렬/집계가 매 배치 | reduce/aggregate가 hot path |
| 스레드풀 사이즈만 키우고 실제 CPU/IO 사용률 미측정 | 메트릭 없음 = 최적화 근거 없음 |
| 공유 캐시/커넥션풀의 대기 시간이 처리 시간보다 큼 | trace/profile에서 "wait for pool" 비율 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "스레드/워커 수를 늘렸는데 속도가 안 나옴" | 순차 구간이 지배적 — Amdahl 상한 도달 |
| "DB 하나 바라보는 워커 100개" | 병목이 DB로 이동했을 뿐 |
| "분산으로 갔는데 latency가 더 증가" | 조율 오버헤드 > 병렬 이득 |
| async로 전환했는데 체감 빠름이 없음 | 실제로는 순차 await 체인 |
| "map/parallel 처리 후 reduce에 시간 대부분" | reduce가 순차 구간 |
| 대부분 요청이 같은 캐시키/리소스로 몰림 | 팬아웃이 무효 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 순차 구간 측정됨, 병렬화 상한 계산됨, 병목이 실제 병렬 구간에 있음 |
| **WARN** | 병렬화했지만 순차 병목(공유 락·중앙 수집점)이 지배적 — 개선 한계 도달 |
| **FAIL** | 측정 없는 병렬화 OR 병렬 블록 내 직렬화 OR 조율 오버헤드 무시 OR "스레드 수만 늘리기" |

## 간단한 상한 계산

| 순차 비율 (1-p) | n=4 | n=16 | n=∞ |
|----------------|-----|------|-----|
| 5% | 3.5× | 9.1× | 20× |
| 10% | 3.1× | 6.4× | 10× |
| 20% | 2.5× | 4.0× | 5× |
| 50% | 1.6× | 1.9× | 2× |

**교훈**: 순차 구간 5%만 있어도 무한 코어로 20배가 한계. "병렬화하면 N배 빨라진다"는 거의 항상 틀린 어림.

## 병목 이동 패턴

```
단일 스레드
  └→ CPU 병목
      └→ 멀티스레드 (Amdahl 적용)
          └→ 락 경합 / 공유 큐 병목
              └→ 파티셔닝
                  └→ DB / 네트워크 병목
                      └→ 샤딩 / 캐싱
                          └→ 캐시 일관성 / 조율 병목 (분산 합의)
                              └→ ...
```

병목은 **사라지지 않고 이동한다**. 최적화는 "어느 병목을 받아들일 것인가"의 선택.

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 병렬 블록 안에 공유 자원 | **파티셔닝** — 자원을 키별로 나눠 경합 제거 |
| Promise.all 안의 순차 await | **진짜 병렬로 재작성** — 독립 I/O 동시 발행 |
| 중앙 큐/DB에 수렴 | **fan-out 이후의 수렴점을 분산**하거나 버퍼/배치 |
| 조율 오버헤드 과다 | **거친 단위(coarse-grained)로 병렬화** — 작업 단위를 키움 |
| 분산 합의 비용 | **일관성 수준 완화** (strong → eventual)로 합의 빈도 감소 |
| "더 많은 워커" 유혹 | **먼저 측정** — 실제 CPU·IO·Lock 대기 비율 |
| reduce가 병목 | **트리 리듀스/결합법칙 이용한 단계별 집계** |

## 측정이 먼저

Amdahl's Law의 가장 큰 교훈은 **측정 없는 병렬화는 무의미**라는 점:

1. 현재 워크로드의 순차/병렬 비율 측정 (프로파일러)
2. 이론적 상한 계산 `S(n) = 1 / ((1-p) + p/n)`
3. 상한이 원하는 수준을 줄 수 있는지 확인
4. 안 되면 알고리즘/구조 변경 (Amdahl 틀 자체를 바꾸기)
5. 된다면 병렬화 후 실측, 오버헤드 감산

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| GOODHARTS-LAW | "스레드 수" 자체를 목표로 삼으면 Goodhart 위반 — 실제 throughput를 봐야 |
| ORTHOGONALITY | 병렬 구간이 실제로 직교(독립)해야 병렬화 효과가 나옴 |
| SoC | 순차 구간과 병렬 구간의 관심사 분리가 있어야 각각 최적화 가능 |
| IDEMPOTENCY | 병렬·재시도 환경에서 idempotent 아니면 결과가 망가진다 |
| LEAKY-ABSTRACTION | async/await·futures 추상화는 성능 경로에서 자주 누수된다 — Amdahl 계산할 땐 누수 인식 필수 |

## 주의

- Amdahl's Law는 "고정 문제 크기" 가정. 문제 크기가 함께 커지면 Gustafson's Law가 더 낙관적 (weak scaling)
- "병렬화 하지 말라"가 아니다. "상한을 인식하고, 측정하고, 순차 구간을 함께 줄여라"
- 네트워크·분산에서는 Amdahl의 "순차 구간"이 **조율 비용**으로 나타남
- 사용자 체감 latency vs 시스템 throughput은 다른 문제 — 병렬화는 보통 throughput에 유리, tail latency에는 복잡
- 마이크로 최적화 전에 Amdahl 상한 계산하고 나면 "이 최적화가 의미 있는 개선인가"가 명확해진다
