# BOY-SCOUT — 보이스카웃 규칙

## 정의

> "Always leave the campground cleaner than you found it."

코드를 수정할 때, 본래 목적 외에 작은 개선을 함께 한다. 거창한 리팩토링이 아니라 **지나가면서 하는 작은 정리**. 시간이 지날수록 코드가 나아지는 방향.

## 적정 범위

Boy Scout Rule의 핵심은 **작은 개선**이다. 범위를 넘으면 scope creep.

### 허용 (1-5분 이내)

- 변수/함수명 명확하게 변경 (작업 중인 파일 내)
- 사용하지 않는 import 제거
- 불필요한 주석 삭제
- 타입 명시 (`any` → 구체 타입)
- 매직 넘버 → 상수 추출
- 중첩 줄이기 (early return)

### 금지 (별도 작업으로 분리)

- 파일 구조 변경 / 파일 이동
- 인터페이스 변경
- 대규모 이름 변경 (여러 파일에 걸침)
- 테스트 추가/수정
- 의존성 변경

## 핵심 판단

- **"이 개선이 현재 커밋의 목적과 무관하게 독립적인가?"** — 독립적이고 작으면 OK
- **"이 개선이 리뷰어를 혼란스럽게 하는가?"** — 혼란스러우면 별도 커밋/PR
- **"5분 안에 끝나는가?"** — 안 끝나면 TODO로 남긴다

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 사용하지 않는 import 방치 | lint 경고 누적 수 |
| `any` 타입 증가 추세 | git log에서 `any` 추가 빈도 |
| TODO 주석 증가 추세 | grep `TODO\|FIXME\|HACK` 개수 변화 |
| lint 경고 수 증가 | CI lint 결과 추세 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "이 코드 지저분한데 안 고치고 그냥 갈게" | Boy Scout 미적용 |
| PR에 본래 목적과 무관한 대규모 정리 포함 | 과도한 적용 (scope creep) |
| 팀 내 "건드리면 안 되는 파일" 존재 | 정리 부채가 쌓인 신호 |
| 매번 같은 파일에서 같은 불편함 반복 | 정리 기회를 놓치고 있음 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 커밋에 작은 정리 포함, lint 경고 감소 추세, TODO 관리됨 |
| **WARN** | 정리 없이 기능만 추가 OR lint 경고 정체 |
| **FAIL** | lint 경고 증가 추세 OR `any`/TODO 누적 OR 정리 PR이 별도로 대량 필요 |

## 실천 체크리스트

파일을 수정할 때 빠르게 확인:

```
□ 사용하지 않는 import가 있는가?
□ 이름이 불명확한 변수/함수가 있는가? (작업 중인 부분)
□ 불필요한 주석이 있는가?
□ any 타입을 구체화할 수 있는가?
□ 매직 넘버가 있는가?
□ early return으로 중첩을 줄일 수 있는가?
```

하나라도 해당하면 함께 정리한다. 단, **현재 작업 파일 내에서만**.

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 정리 범위 판단이 어려움 | **별도 커밋** — 기능 커밋과 정리 커밋을 분리 |
| 팀에서 정리를 안 함 | **lint 자동화** — CI에서 경고 증가 시 실패 |
| 정리할 것이 너무 많음 | **TODO 기록 → 별도 tidy 작업** — Boy Scout이 아닌 계획된 정리 |
| PR 리뷰에서 정리 범위 논쟁 | **팀 합의** — "같은 파일 내 5분 이내"를 기준선으로 |

## 주의

- Boy Scout ≠ 대규모 리팩토링. 범위를 넘기면 본래 작업의 리뷰가 어려워진다
- 팀 컨벤션과 충돌하면 팀 규칙이 우선. 자의적 스타일 변경은 Boy Scout이 아니다
- 테스트가 없는 코드에서 Boy Scout 정리는 위험할 수 있다. 동작 변경이 아닌 것만
- Git blame이 오염된다는 우려가 있으나, 코드 품질이 blame 이력보다 중요하다
