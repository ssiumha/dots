# BROKEN-WINDOWS — 깨진 유리창 이론

## 정의

> "One broken window, left unrepaired, instills in the inhabitants of the building a sense of abandonment. So another window gets broken."

환경이 행동을 결정한다. 코드베이스에 품질 문제가 방치되면 "여기는 그런 곳"이라는 인식이 퍼지고, 추가 품질 저하가 가속된다. **개인의 의지가 아니라 시스템이 품질을 강제해야** 유리창이 깨지지 않는다.

## 핵심 판단

- **"이 코드베이스에 방치된 문제가 있는가?"** — lint 경고, 깨진 테스트, TODO 누적
- **"새 문제 추가를 시스템이 차단하는가?"** — CI, hook, 자동화가 있는가
- **"품질이 시간이 지남에 따라 나아지는가, 악화되는가?"** — 추세가 중요

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| lint 경고 증가 추세 | CI lint 결과 시계열 비교 |
| `@ts-ignore` / `noqa` / `eslint-disable` 증가 | grep 카운트 추세 |
| 깨진 테스트 방치 (skip/pending) | grep `skip\|pending\|xit\|xdescribe` 수 추세 |
| TODO/FIXME/HACK 누적 | grep 카운트 증가 추세 |
| `any` 타입 증가 | grep `any` 카운트 추세 |
| 커버리지 하락 추세 | 커버리지 리포트 시계열 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "이 파일은 원래 이래요" | 방치 정당화 |
| "하나쯤 괜찮겠지" | 깨진 유리창 추가 허용 |
| 팀에서 lint 경고를 무시하는 문화 | 시스템 부재 |
| 코드 리뷰에서 품질 이슈를 안 잡음 | 리뷰 시스템 미작동 |
| "나중에 고치자" 가 실제로 안 고쳐짐 | TODO가 영구화 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | lint 경고 0 유지, 깨진 테스트 0, CI가 품질 저하 차단, 경고 감소 추세 |
| **WARN** | lint 경고 존재하나 증가하지 않음, 일부 skip 테스트 |
| **FAIL** | lint 경고 증가 추세 OR 깨진 테스트 방치 OR CI에서 품질 체크 없음 OR any/ignore 증가 |

## 시스템 강제 수단

개인 습관이 아니라 자동화로 유리창을 보호한다:

| 계층 | 수단 | 차단하는 것 |
|------|------|------------|
| **편집기** | 저장 시 자동 포맷 | 스타일 불일치 |
| **pre-commit** | lint, 타입 체크 | 경고 추가 |
| **CI** | 테스트, 커버리지 게이트 | 테스트 실패, 커버리지 하락 |
| **PR 리뷰** | 자동 리뷰 봇 (lint 결과 코멘트) | 코드 품질 저하 |
| **모니터링** | 경고 수 대시보드 | 추세 악화 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| lint 경고 100개 누적 | **Ratchet** — 현재 수를 기준선으로, 증가만 차단. 점진적 감소 |
| 깨진 테스트 방치 | **Zero Tolerance CI** — 1개라도 실패하면 머지 차단 |
| TODO 누적 | **기한 부여** — `TODO(2026-05-01)` 형태, 기한 초과 시 CI 경고 |
| "나중에 고치자" 문화 | **Boy Scout + CI** — 파일 수정 시 경고 감소 요구 |
| 코드 리뷰에서 품질 안 봄 | **자동화된 품질 체크** — 사람 리뷰 전에 봇이 먼저 |

## 유리창 깨기 vs 유리창 보호하기

```
유리창이 깨지는 과정:
  lint 경고 1개 방치
  → "여기 원래 그래" 인식
  → 경고 10개로 증가
  → "어차피 많으니 하나 더"
  → 경고 100개
  → 아무도 안 봄
  → 코드베이스 부패

유리창을 보호하는 시스템:
  lint 경고 추가 → CI 실패 → 머지 차단
  → 개발자가 수정 후 다시 제출
  → 경고 0 유지
  → "여기는 깨끗한 곳" 인식
  → 자발적으로도 품질 유지
```

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| [[TECHNICAL-DEBT]] | **짝** — Broken Windows는 "방치된 문제 = 부패", Technical Debt은 "의식적 차입은 OK". 무지하게 쌓인 부채가 곧 깨진 유리창 |
| [[SELF-TESTING-CODE]] | flaky 테스트 방치, 커버리지 하락은 깨진 유리창 — Self-Testing은 그것을 빌드 실패로 차단하는 시스템 형태 |
| [[BOY-SCOUT]] | Boy Scout은 개인 실천, Broken Windows는 시스템 강제. 시스템이 더 근본적 |
| [[FAIL-FAST]] | 깨진 유리창을 빨리 감지 (fail fast) + 즉시 차단 |
| [[CHESTERTONS-FENCE]] | 기존 lint 규칙을 "불필요해 보여서" 끄면 유리창이 깨진다 |

## 주의

- 경고 0이 목표가 되면 `eslint-disable`을 남발하게 된다. 이것은 유리창을 판자로 가린 것
- Ratchet 전략은 현실적. 100개 경고를 하루에 0으로 만들 필요 없다. "증가하지 않는다"가 첫 목표
- 팀 합의 없이 혼자 엄격하게 하면 마찰만 생긴다. 시스템 도입은 팀과 함께
- "깨끗한 코드"의 정의는 팀마다 다르다. 중요한 것은 **합의된 기준이 시스템으로 강제되는 것**
