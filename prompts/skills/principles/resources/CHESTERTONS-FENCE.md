# CHESTERTONS-FENCE — Chesterton's Fence

## 정의

> "Don't ever take a fence down until you know the reason why it was put up."

기존 규칙, 제약, 코드가 불합리해 보여도 그것이 존재하는 이유가 있다. 이유를 이해하기 전에 제거하면 그 규칙이 막고 있던 문제가 다시 터진다. **"왜 있는지"를 먼저 밝혀라.**

## 핵심 판단

- **"이 코드/규칙이 왜 존재하는지 설명할 수 있는가?"** — 못하면 아직 제거하면 안 된다
- **"이것을 제거했을 때 무엇이 깨지는지 아는가?"** — 모르면 제거하면 안 된다
- **"git blame / 커밋 메시지로 이유를 찾았는가?"** — 먼저 조사하라

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| `eslint-disable` / `noqa` / `@ts-ignore` 제거 후 빌드 실패 | 주석 제거 → CI 결과 확인 |
| 이상해 보이는 sleep/delay 제거 후 race condition | 타이밍 관련 코드 변경 이력 확인 |
| "dead code" 제거 후 특정 환경에서 장애 | 조건부 실행 코드(feature flag 등) 확인 |
| 오래된 TODO 주석 무단 삭제 | TODO에 연결된 이슈/컨텍스트 확인 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "이 코드 왜 이렇게 짰지? 그냥 고치자" | 이유 조사 없이 변경 |
| "이 규칙 불필요해 보이는데 빼자" | 규칙의 배경 미확인 |
| "이 의존성 안 쓰는 것 같은데 지우자" | 간접 의존/런타임 의존 미확인 |
| "레거시니까 다 지우고 새로 짜자" | 기존 시스템의 학습 무시 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 제거/변경 전 git blame 확인, 이유 기록, 테스트로 안전성 확인 |
| **WARN** | 일부 조사 없이 제거했으나 테스트가 잡아줌 |
| **FAIL** | 이유 모르고 제거 → 장애 OR 조사 없이 "불필요해 보여서" 삭제 반복 |

## 조사 프로토콜

코드/규칙을 제거하기 전 체크리스트:

```
1. git blame — 누가, 언제 추가했는가?
2. 커밋 메시지 — 왜 추가했는가?
3. PR/이슈 — 관련 논의가 있는가?
4. 주석 — 이유가 적혀 있는가?
5. 테스트 — 이것을 보호하는 테스트가 있는가?
6. 팀원에게 질문 — 기억하는 사람이 있는가?

모든 조사를 했는데도 이유를 모르면:
→ 제거하되 **되돌릴 수 있게** (별도 커밋, feature flag)
→ 제거 이유를 커밋 메시지에 기록
→ 모니터링 강화
```

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 이유를 모르는 코드를 발견 | **조사 먼저** — git blame, 커밋 히스토리, PR 확인 |
| 이유를 알고 더 이상 유효하지 않음 | **제거 + 이유 기록** — 커밋 메시지에 "왜 제거했는지" 명시 |
| 레거시 교체 | **Strangler Fig** — 한 번에 교체가 아니라 점진적으로 |
| 울타리를 세울 때 | **이유를 기록** — 미래의 누군가를 위해 주석/커밋 메시지에 배경 설명 |

## 울타리를 세우는 쪽의 책임

Chesterton's Fence는 제거하는 쪽뿐 아니라 **세우는 쪽에도 의무**를 부과한다:

- `eslint-disable`을 쓸 때 **왜 필요한지** 주석 작성
- 이상한 workaround를 넣을 때 **어떤 문제를 피하는지** 커밋 메시지에 기록
- 특이한 제약을 추가할 때 **관련 이슈/티켓 번호** 연결

이유 없는 울타리는 미래에 반드시 잘못 제거된다.

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| [[COMMENT-WHY]] | **짝** — Fence는 "이유 모르면 치우지 마라" (제거자 책임), Comment-WHY는 "미래의 누군가가 이유를 알 수 있게 남겨라" (작성자 책임). 두 원칙이 함께 작동해야 의도가 보존됨 |
| [[BOY-SCOUT]] | Boy Scout은 "정리하라"지만, Chesterton's Fence는 "이해 먼저" — 충돌 시 Fence 우선 |
| [[YAGNI]] | YAGNI는 "불필요하면 안 만든다", Fence는 "불필요해 보여도 이유를 먼저 확인" |
| [[GALLS-LAW]] | 기존 시스템이 이유 있이 진화한 것 — Fence는 그 진화의 각 단계를 존중하라는 것 |

## 주의

- Chesterton's Fence ≠ "아무것도 바꾸지 마라". 이유를 확인한 후에는 과감히 제거해도 된다
- 이유를 아는데도 "원래 있던 거니까" 유지하는 것은 반대 방향의 오류
- 팀에 오래된 멤버가 없으면 울타리의 이유를 알기 어렵다 — 이것이 **문서화와 커밋 메시지**가 중요한 이유
- 때로는 울타리가 진짜 실수로 세워진 것일 수 있다. 조사 후 확신이 서면 제거
