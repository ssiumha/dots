# COHESION-COUPLING — 응집도와 결합도

## 정의

> "Maximize cohesion within modules, minimize coupling between modules."

소프트웨어 설계의 가장 기본적인 측정 축. 다른 원칙들(SRP, SoC, LoD, DIP)이 각각 부분적으로 다루지만, 이 원칙은 **명시적 측정 기준**을 제공한다.

- **Cohesion(응집도)** — 모듈 내부 요소들이 얼마나 하나의 목적에 집중하는가
- **Coupling(결합도)** — 모듈 간에 얼마나 서로 의존하는가

## 응집도 스펙트럼 (높을수록 좋다)

```
나쁨 ──────────────────────────────────── 좋음
우연적 → 논리적 → 시간적 → 절차적 → 통신적 → 순차적 → 기능적
```

| 등급 | 설명 | 예시 |
|------|------|------|
| **기능적** (최고) | 모든 요소가 하나의 기능 수행 | `calculateTax()` |
| **순차적** | 한 요소의 출력이 다음 입력 | 파이프라인 스테이지 |
| **통신적** | 같은 데이터를 다루는 요소 모음 | 같은 엔티티의 CRUD |
| **시간적** | 같은 시점에 실행되는 것 모음 | `init()`, `startup()` |
| **논리적** | 같은 카테고리지만 관련 없음 | `StringUtils` (trim, pad, parse 혼재) |
| **우연적** (최악) | 아무 관련 없는 것 모음 | `Helpers`, `Miscellaneous` |

## 결합도 스펙트럼 (낮을수록 좋다)

```
좋음 ──────────────────────────────────── 나쁨
메시지 → 데이터 → 스탬프 → 제어 → 외부 → 공통 → 내용
```

| 등급 | 설명 | 예시 |
|------|------|------|
| **메시지** (최저) | 메서드 호출만, 파라미터 없음 | `notify()` |
| **데이터** | 단순 데이터만 전달 | `calculate(price, qty)` |
| **스탬프** | 구조체/객체 전달 (일부만 사용) | `process(order)` 중 `order.id`만 사용 |
| **제어** | 플래그로 상대 동작을 지시 | `format(data, isHTML=true)` |
| **공통** | 전역 변수/상태 공유 | 싱글톤 상태, 전역 config 직접 참조 |
| **내용** (최악) | 상대 내부를 직접 접근/수정 | `module._privateField = 42` |

## 핵심 판단

- **"이 모듈의 이름을 한 단어로 말할 수 있는가?"** — 못하면 응집도 낮음
- **"이 모듈을 변경하면 다른 모듈을 몇 개나 고쳐야 하는가?"** — 많으면 결합도 높음
- **"이 모듈을 독립적으로 테스트할 수 있는가?"** — 못하면 결합도 높음

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 파일당 import 수 과다 | import 수 카운트 (>15이면 결합 의심) |
| 한 파일에서 여러 도메인 import | import 경로의 도메인 다양도 |
| 순환 의존 | 의존성 그래프에서 cycle 검출 |
| 전역 상태 참조 | grep `global\|window\.\|process\.env` (인프라 경계 외) |
| Fan-out 과다 (이 모듈이 의존하는 수) | import/require 카운트 |
| Fan-in 과다 (이 모듈에 의존하는 수) | 역참조 카운트 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "이 클래스 이름에 And가 들어감" | 응집도 낮음 (두 가지 역할) |
| 변경 시 관련 없어 보이는 테스트가 깨짐 | 숨겨진 결합 |
| 모듈 이름이 Manager/Handler/Service 등 모호 | 응집도 불명확 |
| 리팩토링하려면 먼저 다른 모듈부터 건드려야 함 | 높은 결합 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 모듈별 단일 책임 명확, import 15개 이하, 순환 의존 0, 독립 테스트 가능 |
| **WARN** | import 15-25개 OR 순환 의존 1-2개 OR 일부 God class |
| **FAIL** | import 25개+ OR 순환 의존 3개+ OR 변경 시 연쇄 수정 필수 |

## 측정 도구

| 메트릭 | 공식 | 의미 |
|--------|------|------|
| **Instability (I)** | Ce / (Ca + Ce) | 0=안정, 1=불안정. 의존은 I가 낮은 쪽을 향해야 |
| **Abstractness (A)** | 추상 타입 수 / 전체 타입 수 | 0=구체, 1=추상 |
| **Distance (D)** | \|A + I - 1\| | 0에 가까울수록 균형. Main Sequence에서의 거리 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 우연적/논리적 응집 (Utils, Helpers) | **도메인별 분리** — 사용하는 쪽 근처로 이동 |
| 제어 결합 (플래그 파라미터) | **Strategy 패턴** 또는 별도 함수 분리 |
| 순환 의존 | **의존 역전 (DIP)** 또는 **이벤트 기반** 분리 |
| 공통 결합 (전역 상태) | **의존성 주입** — 전역 접근 대신 명시적 전달 |
| 내용 결합 (내부 직접 접근) | **ENCAPSULATION** — private + 공개 인터페이스 |

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| SRP | 응집도의 클래스 레벨 적용 |
| SoC | 응집도의 아키텍처 레벨 적용 |
| LoD | 결합도 제한의 호출 레벨 규칙 |
| DIP | 결합 방향 역전을 통한 결합도 감소 |
| ENCAPSULATION | 내용 결합 방지의 직접적 수단 |

## 주의

- 결합도 0은 불가능하고 바람직하지도 않다. 모듈은 협력해야 한다. 목표는 **불필요한** 결합 제거
- 높은 응집이 항상 작은 모듈을 의미하지 않는다. 하나의 복잡한 기능은 큰 모듈이 될 수 있다
- 간접화(인터페이스, 이벤트)로 결합을 숨기면 결합도는 낮아지지만 이해 비용이 올라간다. 트레이드오프
- Utils/Helpers 클래스를 무조건 나쁘다고 할 수 없다. 순수 함수 유틸은 결합도가 낮다. 문제는 상태를 가진 God class
