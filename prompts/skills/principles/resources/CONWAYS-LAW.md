# CONWAYS-LAW — Conway's Law

## 정의

> "Organizations which design systems are constrained to produce designs which are copies of the communication structures of these organizations."

소프트웨어의 모듈 경계는 결국 그 소프트웨어를 만든 사람들이 **누구와 자주 대화하는가**의 반영이다. 네 팀이 작업하면 네 덩어리짜리 컴파일러가 나온다. 이 현상은 피할 수 없고, **거스르려 하면 구조가 뒤틀린다**. 반대로 원하는 구조가 있으면 그 모양대로 조직을 먼저 맞추는 것이 Inverse Conway Maneuver.

## 핵심 판단

- **"이 모듈 경계가 실제 커뮤니케이션 경계와 일치하는가?"** — 다르면 경계가 깨지기 쉬움
- **"한 모듈을 여러 팀이 동시에 건드리는가?"** — 그렇다면 Conway 미스매치
- **"팀 간 조율이 잦은 PR이 계속 나오는가?"** — 경계가 잘못 그어졌다는 신호
- **"원하는 아키텍처 모양에 맞게 조직이 정렬되어 있는가?"** — 아니면 구조가 조직 모양으로 되돌아간다

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 한 모듈/파일에 서로 다른 팀 저자가 혼재 | `git log --format="%an" <path>` → 저자 분포, CODEOWNERS와 대조 |
| 같은 feature가 cross-team PR로 반복 | `git log --oneline` → PR 제목의 팀 태그/라벨 분포 |
| CODEOWNERS 경계가 디렉토리 구조와 불일치 | CODEOWNERS 경로 vs 실제 디렉토리 depth |
| 두 팀이 같은 테이블/스키마를 쓰는 서비스 | `grep -r "TableName"` → 여러 서비스 디렉토리에서 참조 |
| 인접 모듈로의 import가 비대칭 | 모듈 간 의존 그래프의 cross-boundary edge 수 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "이 기능 하나 추가에 3팀 승인이 필요함" | 경계가 기능 흐름과 직교 |
| 한 서비스인데 팀 A가 API, 팀 B가 DB를 소유 | 책임 분할이 수직이 아닌 수평 → 변경마다 조율 |
| 팀 분리 후에도 이전 모놀리스 구조가 남음 | Conway's Law 역작용을 외면한 상태 |
| "이 부분은 어느 팀 책임?"이 자주 애매 | 조직-코드 매핑 부재 |
| 공유 라이브러리에 팀별 특화 로직이 누적 | 경계 없이 commons에 떠넘김 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 모듈 경계 = 팀 경계, CODEOWNERS 명확, cross-team 수정이 드물고 명시적 |
| **WARN** | 일부 공유 모듈이 cross-team이지만 소유자가 문서화됨 |
| **FAIL** | 한 모듈 다수 팀 상시 수정 OR CODEOWNERS 부재 OR 팀 분리와 코드 구조가 정반대 |

## Inverse Conway Maneuver

구조를 바꾸고 싶으면 **조직을 먼저 바꾼다**:

```
원하는 아키텍처 (예: 도메인 별 독립 서비스)
  ↑ 구조가 이 모양으로 나오려면
조직의 커뮤니케이션 라인이 그 모양이어야 한다
  ↑ 즉
팀 분할 · 책임 경계 · 소통 채널을 원하는 아키텍처 모양으로 먼저 정렬
```

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 마이크로서비스를 원하지만 한 팀이 전부 만듦 | 서비스별 소유팀 분리 후 분할 — 조직이 먼저 |
| 모놀리스 → 모듈러 모놀리스 전환 | CODEOWNERS로 경계 먼저 그어 소유권부터 나눔 |
| 공유 라이브러리가 진흙탕 | 각 모듈의 소유자 지정, 외부 팀은 PR로만 기여 |
| 한 도메인을 2팀이 나눠 가짐 | 도메인 주도로 재편 — 둘 중 하나가 전담 |
| 신규 feature 경계 모호 | 설계 단계에서 "어느 팀이 유지보수할 것인가"를 먼저 결정 |

## 주의

- Conway's Law는 "해야 할 일"이 아니라 "일어나는 일"이다. 거스르려 하면 드리프트로 되돌아온다
- 원격/비동기 조직은 Conway의 "커뮤니케이션 구조"가 Slack 채널·meeting 구조로 나타난다. 코드 경계는 이 비공식 경계도 따라간다
- 조직 개편이 잦으면 구조도 계속 흔들린다 — 변경 비용을 인식
- 소규모 팀(1~3명)에서는 Conway's Law가 약하게 작동. 조직이 커질수록 지배적
- "좋은 아키텍처를 만들자" 전에 "이 조직에서 유지 가능한 아키텍처인가"를 먼저 물어야 한다
