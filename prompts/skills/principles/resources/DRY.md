# DRY — Don't Repeat Yourself

## 정의

> "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."

DRY는 코드 중복만이 아니다. 지식의 중복이다.
코드, 문서, 스키마, 설정 — 어디든 같은 지식이 2곳에 있으면 DRY 위반.

## 핵심 판단

- **"이 지식이 다른 곳에도 있는가?"** — 있다면 Single Source of Truth를 정한다
- **"이 두 코드가 같은 이유로 같은가?"** — 우연한 유사성은 중복이 아니다
- **"하나를 바꾸면 다른 것도 바꿔야 하는가?"** — 그렇다면 진짜 중복

## 우연한 유사성 vs 진짜 중복

| | 진짜 중복 | 우연한 유사성 |
|---|---|---|
| 변경 이유 | 같다 | 다르다 |
| 하나를 바꾸면 | 다른 것도 바꿔야 | 바꿀 필요 없음 |
| 예시 | 두 곳의 이메일 검증 | 주문 검증과 유저 검증이 비슷 |
| 대응 | Extract → 공유 | 그대로 둔다 |

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 임계값 | 검증 방법 |
|------|--------|-----------|
| 동일/유사 코드 블록 | 3줄+ 동일 블록 2곳+ | jscpd, PMD CPD, 클론 탐지 도구 |
| 상수/매직넘버 중복 정의 | 같은 값 2곳+ | grep으로 동일 리터럴 탐색 |
| 동일 정규식 패턴 반복 | 2곳+ | grep 정규식 리터럴 |
| 동일 SQL 쿼리/조건 반복 | 2곳+ | grep SQL 패턴 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| 코드와 문서의 불일치 | 같은 지식이 동기화 안 됨 |
| API 스펙과 구현의 수동 동기화 | OpenAPI ↔ 코드가 별도 관리 |
| 테스트 fixture와 시드 데이터 중복 | 같은 데이터 구조가 여러 형태로 |
| 유사한 로직이 다른 언어/레이어에 | 프론트/백엔드 동일 검증 로직 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 동일 코드 블록 중복 0, 상수 중복 0 |
| **WARN** | 동일 코드 블록 1-2개 OR 상수 중복 1-3개 |
| **FAIL** | 동일 코드 블록 3개+ OR 코드-문서 불일치 OR 수동 동기화 필요 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 동일 코드 블록 | **Extract Method/Function** |
| 상수 중복 | **상수 파일/enum으로 집중화** |
| 코드-문서 동기화 | **코드에서 문서 자동 생성** |
| 프론트/백 동일 검증 | **공유 스키마** (JSON Schema, Zod) |
| 설정 중복 | **Single Source of Truth + 파생** |

## Rule of Three

DRY의 실용적 적용 — 같은 것이 3번 반복되면 그때 추상화한다:
- 1번: 그냥 작성
- 2번: 메모 (중복 인지)
- 3번: 추출 (추상화)

2번까지는 우연의 일치일 수 있다. 3번이면 패턴이다.

## 주의

- DRY는 코드 줄 수를 줄이는 것이 아니다. **지식의 단일 표현**이 핵심
- 우연한 유사성을 억지로 합치면 오히려 결합도만 높아진다
- 과도한 DRY → 추상화 폭발 → KISS 위반. Rule of Three로 균형
- 때로는 약간의 중복이 잘못된 추상화보다 낫다
