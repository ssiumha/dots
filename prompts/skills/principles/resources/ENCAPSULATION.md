# ENCAPSULATION — 캡슐화 (정보 은닉)

## 정의

> "모듈은 구현 세부사항을 숨기고, 안정적인 인터페이스만 외부에 노출해야 한다."

LoD가 "호출 규칙"이라면, 캡슐화는 "노출 범위"에 대한 원칙이다.
내부가 바뀌어도 외부가 영향받지 않아야 한다.

## 핵심 판단

- **"이 모듈의 내부 구현을 바꿔도 호출자가 모르는가?"** — 모르면 잘 캡슐화됨
- **"외부에서 내부 상태를 직접 조작할 수 있는가?"** — 조작 가능하면 위반
- **"이 모듈이 노출하는 것 중 내부 세부사항은 없는가?"** — 있으면 위반

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| public 필드 (getter/setter 없이 직접 접근) | ast-grep: public field 탐지 |
| 내부 자료구조 직접 반환 (mutable) | 리스트/딕셔너리 반환 시 복사 여부 확인 |
| `_private` 필드의 외부 접근 | grep `\._[a-z]` in 다른 모듈 |
| 모든 필드에 getter/setter 존재 (Anemic Model) | 클래스의 public 메서드 중 get/set 비율 |
| 구현 타입을 반환 (ArrayList 대신 List) | 반환 타입 vs 인터페이스 타입 비교 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| 내부 상태를 꺼내서 외부에서 처리 | 로직이 데이터 밖에 있음 (Feature Envy) |
| 구현 변경 시 호출자도 변경 필요 | 인터페이스가 구현에 결합 |
| 모듈의 "사용법"을 알려면 내부를 읽어야 | 인터페이스만으로 사용 불가 |
| 테스트가 내부 상태를 직접 검증 | `assert obj._internal == 42` |
| 설정/상수가 전역으로 노출 | 모듈 내부 설정이 외부에서 변경 가능 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | public 필드 0, 내부 상태 외부 미노출, 인터페이스만으로 사용 가능 |
| **WARN** | public 필드 1-2개 OR getter/setter 과다 (Anemic Model 경향) |
| **FAIL** | 내부 상태 직접 조작 OR 구현 변경 시 호출자 변경 필요 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| public 필드 | **private + 메서드** — 행동을 제공, 데이터를 숨김 |
| mutable 내부 반환 | **방어적 복사 또는 불변 객체** 반환 |
| getter/setter 남용 | **Tell Don't Ask** — 데이터를 꺼내지 말고 객체에게 시킴 |
| 구현 타입 노출 | **인터페이스 타입 반환** — `List` not `ArrayList` |
| 전역 설정 | **모듈 내 private 설정** + 필요 시 주입 |

## LoD와의 관계

| | LoD | Encapsulation |
|---|---|---|
| 관점 | 호출 규칙 (누구에게 말할 수 있는가) | 노출 범위 (무엇을 보여줄 것인가) |
| 위반 | `a.b.c.d()` 체이닝 | public 필드, 내부 상태 노출 |
| 해결 | 위임 메서드 추가 | private + 인터페이스 |

둘 다 지키면 모듈 간 결합도가 크게 낮아진다.

## 주의

- 캡슐화 ≠ 모든 것을 private으로. 의미 있는 인터페이스는 적극적으로 노출
- DTO/Value Object는 예외 — 순수 데이터 컨테이너는 public 필드가 자연스러움
- 언어마다 관례가 다름: Python은 `_` 컨벤션, Go는 대문자/소문자, JS/TS는 `#`/`private`
- 테스트를 위해 캡슐화를 깨는 것은 설계 문제의 신호 — 테스트 가능한 설계를 먼저 고민
