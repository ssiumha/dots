# GOODHARTS-LAW — Goodhart's Law

## 정의

> "When a measure becomes a target, it ceases to be a good measure."

지표는 현상을 **관찰**하기 위한 도구다. 그런데 그 지표를 **목표**로 설정하면, 사람들은 현상을 개선하는 대신 지표 자체를 최적화하기 시작한다. 결과적으로 지표는 올라가지만 실제 품질은 제자리거나 오히려 나빠진다.

## 핵심 판단

- **"이 지표를 올리는 것이 진짜 품질 향상과 같은가?"** — 다르면 Goodhart 위험
- **"지표를 게임할 수 있는 쉬운 방법이 있는가?"** — 있으면 반드시 게임된다
- **"지표가 아닌 결과(outcome)를 보고 있는가?"** — 지표만 보면 위험

## 소프트웨어 개발에서의 대표 사례

| 지표 | 목표로 설정하면 | 실제로 일어나는 일 |
|------|-----------------|-------------------|
| 커버리지 80% | 커버리지 달성이 목표 | assert 없는 테스트, 의미 없는 행 커버 |
| velocity 30pt/sprint | 포인트 달성이 목표 | 스토리 포인트 인플레이션, 작은 일에 큰 포인트 |
| PR 리뷰 시간 < 24h | 빠른 리뷰가 목표 | 대충 LGTM, 실질적 리뷰 없음 |
| 버그 수 0 | 버그 제로가 목표 | 버그를 리포팅하지 않음, "feature"로 재분류 |
| 배포 횟수 | 잦은 배포가 목표 | 의미 없는 변경도 배포, config 변경 배포 카운트 |
| 코드 줄 수 | 생산성 측정 | 불필요한 코드, 복붙, 장황한 구현 |
| PR 크기 < 200줄 | 작은 PR이 목표 | 의미 없이 쪼갠 PR, 문맥 분절 |

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 커버리지는 높은데 mutation testing 통과율 낮음 | mutation testing 결과 비교 |
| assert 없는 테스트 존재 | grep 테스트 파일에서 assert/expect 부재 |
| 지표는 개선되는데 장애/버그는 줄지 않음 | 지표 추세 vs 실제 결과 추세 비교 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "커버리지 채우기 위해" 테스트 작성 | 수단과 목적 전도 |
| 지표 달성을 위한 편법이 공공연한 관행 | 게임이 정상화됨 |
| 팀이 지표에 대해 냉소적 | "어차피 의미 없는데" |
| 관리자가 지표만 보고 판단 | 맥락 없는 최적화 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 지표는 관찰용, 목표는 outcome 기반, 지표 게임 징후 없음 |
| **WARN** | 일부 지표가 목표화됨, 게임 징후 초기 단계 |
| **FAIL** | 지표 달성이 목표, 게임이 공공연함, 지표와 실제 품질 괴리 |

## 올바른 지표 운용

```
❌ 나쁜 운용:
  커버리지 80% 미만 → CI 실패 (목표화)
  → 개발자가 assert 없는 테스트로 80% 채움
  → 커버리지 80%인데 버그는 그대로

✅ 좋은 운용:
  커버리지를 관찰 (대시보드에 표시)
  + mutation testing으로 테스트 품질도 관찰
  + 커버리지 "하락"만 CI에서 차단 (ratchet)
  → 개발자가 의미 있는 테스트를 점진적으로 추가
```

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 단일 지표를 목표로 설정 | **복합 지표** — 여러 지표를 함께 관찰하여 게임 어렵게 |
| 지표 게임이 발생 | **Outcome 기반 목표** — 지표 대신 결과를 목표로 (장애 감소, 고객 만족) |
| 커버리지 목표화 | **Ratchet** — 절대 수치 대신 "하락하지 않기"만 강제 |
| 지표에 대한 냉소 | **지표 폐기 or 재설계** — 의미 없는 지표는 제거 |
| velocity 인플레이션 | **velocity는 예측용** — 목표가 아니라 계획 도구로만 사용 |

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| BROKEN-WINDOWS | 깨진 유리창을 지표(lint 경고 수)로 관리할 때 Goodhart 주의 |
| BOY-SCOUT | "정리 커밋 수"를 지표로 만들면 Goodhart 위반 |
| FAIL-FAST | 감지 시스템의 지표가 게임되면 fail-fast가 무력화 |

## 주의

- Goodhart's Law ≠ "지표를 사용하지 마라". 지표는 필수. **목표로 만들지 마라**
- CI 게이트(커버리지 하락 차단 등)는 목표화가 아니라 ratchet — 이것은 괜찮다
- 팀이 지표를 자발적으로 개선하는 것과 지표 달성을 강제하는 것은 다르다
- 가장 좋은 지표는 **게임하기 어렵고, 관찰하기 쉬운** 것
