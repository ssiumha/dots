# HYRUMS-LAW — Hyrum's Law

## 정의

> "With a sufficient number of users of an API, it does not matter what you promise in the contract: all observable behaviors of your system will be depended upon by somebody."

명시적 계약(DbC)만으로는 충분하지 않다. 에러 메시지 문자열, 응답 필드 순서, 타이밍, 심지어 버그까지 — 관찰 가능한 모든 것이 사실상의 계약이 된다.

## 핵심 판단

- **"이 변경이 명세 밖의 동작을 바꾸는가?"** — 바꾸면 누군가 깨진다
- **"이 API에 사용자가 충분히 많은가?"** — 많을수록 암묵적 의존 확률 높음
- **"관찰 가능하지만 보장하지 않는 동작이 있는가?"** — 있으면 사실상 보장 중

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 에러 메시지 문자열 변경 | grep으로 소비자 측 에러 메시지 파싱 패턴 확인 |
| JSON 응답 필드 순서 변경 | 직렬화 설정에서 ordered/unordered 확인 |
| HTTP 상태 코드 변경 (200 → 201 등) | 소비자 측 상태 코드 분기 확인 |
| 응답 시간 특성 변경 | 소비자 측 타임아웃 설정 확인 |
| 내부 구현 변경으로 부수효과 변경 | 통합 테스트에서 부수효과 검증 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "스펙에 없으니 바꿔도 되겠지" | Hyrum's Law의 핵심 위반 |
| "내부 리팩토링이니 외부 영향 없다" | 관찰 가능한 동작이 바뀔 수 있음 |
| "버그 수정인데 왜 클라이언트가 깨지지?" | 버그에 의존하는 소비자 |
| 문서화되지 않은 API 동작을 활용하는 코드 | 의존하는 쪽도 위험 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | API 변경 시 하위 호환성 검토, 소비자 영향 분석, 관찰 가능 동작 문서화 |
| **WARN** | 명세 내 변경은 검토하나 암묵적 동작 변경은 미검토 |
| **FAIL** | "스펙에 없으니 OK" 태도 OR 소비자 영향 분석 없이 변경 OR 에러 메시지 무단 변경 |

## 관찰 가능한 동작의 범위

```
명시적 계약 (문서화된 것):
  - API 시그니처, 파라미터, 반환 타입
  - 에러 코드, HTTP 상태 코드
  - 문서화된 동작

암묵적 계약 (Hyrum's Law 영역):
  - 에러 메시지 문자열
  - 응답 필드 순서
  - 응답 시간 / 레이턴시 특성
  - 부수효과의 순서
  - null vs 빈 배열 vs 필드 부재
  - 로그 출력 형식
  - 버그의 동작
```

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 에러 메시지에 의존하는 소비자 | **에러 코드 도입** — 구조화된 에러 (code + message 분리) |
| 응답 순서에 의존 | **명시적 정렬** — 순서를 보장하거나 문서에 "보장 안 함" 명시 |
| 내부 리팩토링 시 암묵적 동작 변경 | **소비자 계약 테스트 (Consumer-Driven Contract Test)** |
| 새 API 설계 시 | **관찰 가능 표면 최소화** — 필요한 것만 노출, 나머지 숨김 |
| 비호환 변경이 필요할 때 | **버전 관리** — v1/v2 병행, 마이그레이션 기간 제공 |

## 방어 전략

Hyrum's Law를 완전히 피할 수는 없지만 영향을 줄일 수 있다:

| 전략 | 설명 |
|------|------|
| **관찰 가능 표면 최소화** | 반환하는 데이터, 헤더, 사이드이펙트를 최소한으로 |
| **암묵적 → 명시적** | 보장하는 동작은 문서화, 보장 안 하는 것은 명시적으로 "이건 바뀔 수 있음" |
| **Contract Test** | 소비자가 실제로 의존하는 동작을 테스트로 잡기 |
| **Semantic Versioning** | 관찰 가능한 변경이면 minor/major 버전업 |
| **Canary Release** | 전체 배포 전 소수 사용자에게 먼저 |

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| DbC | DbC는 명시적 계약. Hyrum's Law는 "명시적 계약만으로는 부족하다" |
| POSTELS-LAW | 수신 측이 관대하면 Hyrum's Law의 영향을 줄일 수 있다 |
| ENCAPSULATION | 내부를 잘 숨기면 관찰 가능 표면이 줄어든다 |
| LSP | 하위 타입이 관찰 가능한 동작까지 보존해야 한다 |

## 주의

- Hyrum's Law는 "변경하지 마라"가 아니다. "변경의 영향 범위가 명세보다 넓다"는 인식
- 사용자가 적으면 (내부 API, 소수 소비자) 영향이 적다. 사용자 수에 비례
- 소비자가 암묵적 동작에 의존하는 것은 소비자의 잘못이기도 하지만, 현실적으로 API 제공자가 관리해야
- 오픈소스/퍼블릭 API일수록 Hyrum's Law 영향이 극대화된다
