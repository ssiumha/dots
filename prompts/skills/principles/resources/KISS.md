# KISS — Keep It Simple, Stupid

## 정의

> "가장 단순한 해결책이 거의 항상 최선이다."

단순함은 게으름이 아니라 설계 역량이다.
복잡한 것을 만드는 건 쉽고, 단순한 것을 만드는 게 어렵다.

## 핵심 판단

- **"더 단순하게 할 수 있는가?"** — 3번 자문
- **"새 팀원이 10분 안에 이해할 수 있는가?"** — 못 하면 복잡하다
- **"이 추상화가 지금 필요한가?"** — 미래를 위한 추상화는 YAGNI 위반이기도 하다

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 임계값 | 검증 방법 |
|------|--------|-----------|
| 순환 복잡도 (Cyclomatic Complexity) | > 10 | 정적 분석 도구 (radon, eslint) |
| 중첩 깊이 (nesting depth) | > 3단 | ast-grep, indent 분석 |
| 함수 길이 | > 50줄 | `wc -l` |
| 추상화 계층 수 (파일→호출→실제 로직) | > 4단 | import/call 그래프 추적 |
| 제네릭/타입 파라미터 수 | > 3개 | ast-grep |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "영리한" 코드 | 한 줄에 3개 이상의 연산, 트릭성 코드 |
| 패턴을 위한 패턴 | Factory의 Factory, Strategy의 Strategy |
| 간접 참조 과다 | 로직을 찾으려면 5개 파일을 따라가야 함 |
| 과도한 설정/옵션 | 사용되지 않는 플래그, 설정 가능한 모든 것 |
| "이게 왜 이렇게 되어있지?" | 코드를 읽는 사람이 이유를 모름 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 순환 복잡도 <10, 중첩 <4, 불필요한 추상화 없음 |
| **WARN** | 순환 복잡도 10-15 OR 중첩 4단 OR 간접 참조 과다 |
| **FAIL** | 순환 복잡도 >15 OR 중첩 5단+ OR "이해 불가" 코드 존재 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 깊은 중첩 | **Guard Clause** — 조기 반환으로 중첩 제거 |
| 긴 함수 | **Extract Method** — 의미 단위로 분리 |
| 과도한 추상화 | **Inline** — 한 곳에서만 쓰이는 추상화 제거 |
| 복잡한 조건문 | **Explaining Variable** — 의미 있는 이름의 변수로 분리 |
| 설정 폭발 | **Convention over Configuration** — 합리적 기본값 |

## 주의

- 단순함 ≠ 쉬움. 단순한 설계가 구현은 더 어려울 수 있다
- 단순함 ≠ 기능 부족. 필요한 기능을 빠뜨리는 것은 단순함이 아니라 미완성
- 도메인 자체가 복잡하면 코드도 복잡할 수 있다. 핵심은 "불필요한 복잡도" 제거
