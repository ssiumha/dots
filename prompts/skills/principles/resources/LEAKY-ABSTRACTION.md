# LEAKY-ABSTRACTION — Law of Leaky Abstractions

## 정의

> "All non-trivial abstractions, to some degree, are leaky."

추상화는 복잡성을 가리기 위해 존재하지만, **완벽하게 가릴 수는 없다**. 아래 레이어가 고장나거나, 성능이 이상하거나, 엣지 케이스가 나타나는 순간 사용자는 내부 구현을 알아야 한다. ORM은 SQL을 가리려 하지만 N+1이 터지는 순간 SQL을 알아야 한다. TCP는 "신뢰할 수 있는 스트림"을 추상화하지만 네트워크가 끊기면 신뢰할 수 없어진다.

**추상화는 공짜가 아니다** — 학습 부담, 성능, 디버깅 가능성을 대가로 편의를 얻는 거래.

## 핵심 판단

- **"이 추상화를 쓰는 사람이 내부 구현을 몰라도 일을 마칠 수 있는가?"** — 아니면 이 추상화는 비용만큼의 가치가 없다
- **"추상화의 엣지 케이스를 만났을 때, 내부를 볼 방법이 있는가?"** — 불투명하면 누수가 치명적으로 작용
- **"추상화 때문에 하위 레이어가 무엇을 하는지 모호해지는가?"** — 특히 성능/IO/에러에서
- **"두 레이어의 오류/상태 모델이 일치하는가?"** — 불일치가 크면 누수 빈발

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 추상화 인터페이스에 하위 개념 유출 | Repository 시그니처에 `sql:`/`query_hint:` 등 — grep |
| 일반 인터페이스에 driver-specific 옵션 | HTTP 클라이언트에 특정 헤더 이름 하드코딩 |
| 추상 타입이지만 소비자가 구체 타입 체크 | `isinstance(x, PostgresConn)` 같은 downcast 빈출 |
| 추상화 레이어에서 원시 에러 그대로 throw | `raise` 시 하위 라이브러리 예외 타입 노출 |
| 성능을 위해 추상화 우회하는 "escape hatch" 만연 | `.raw_execute()`, `.native_query()` 사용처 수 |
| 추상화 상수/enum이 하위 구현 이름을 그대로 | `Status.MYSQL_LOCK_TIMEOUT` 같은 이름 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "이 ORM 느린데 SQL 찍어봐야 함" | N+1, 불필요한 JOIN — 추상화가 성능을 가리면서 누수 |
| "이 추상화는 좋아 보이는데 쓰다 보면 결국 내부를 봐야 함" | 비용이 이득을 초과 |
| "레이어 추가했는데 디버깅이 더 어려워짐" | 투명하지 않은 추상화 |
| 다형성 인터페이스지만 실제로는 구현 하나만 존재 | 아직 추상화 필요 근거 부족 (ROT와 연결) |
| 같은 개념을 3~4번 감싸는 래퍼 체인 | 매 레이어가 누수되어 문제 분리 불가 |
| "라이브러리 바꾸려면 어차피 전부 다시 짜야 함" | 추상화가 포팅 가치를 제공 못 함 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 추상화 비용 대비 가치가 명확, 누수 지점이 문서화, escape hatch가 의도적·명시적 |
| **WARN** | 추상화가 대체로 동작하나 일부 성능/에러 경로에서 내부를 알아야 함 |
| **FAIL** | 추상화가 얇아 하위 개념이 상시 노출 OR 추상화가 두꺼운데 디버깅 불가 OR escape hatch가 사실상 유일한 경로 |

## 추상화의 대표 누수 지점

```
정상 경로 (숨겨짐):
  사용자 → 추상 API → 내부 구현

누수 경로 (드러남):
  - 성능: N+1, 캐시 무효화, 과도한 직렬화
  - 에러: 하위 레이어 예외, 네트워크 단절, 타임아웃
  - 상태: 트랜잭션 경계, 격리 수준, 재시도 시 중복
  - 타이밍: async boundary, 이벤트 순서, race condition
  - 리소스: connection pool 고갈, 파일 핸들, 메모리
  - 버전: 하위 라이브러리 bump 시 동작 변화
```

## 개선 패턴

| 상황 | 적용 |
|------|------|
| ORM 성능 문제 | **투명성 제공** — 생성된 SQL 로깅, `explain` hook, N+1 감지 |
| 추상화 escape hatch가 필요한 경우 | **명시적 escape hatch** — `.raw()`·`.unsafe()` 처럼 눈에 띄게 |
| 모든 세부사항을 숨기려는 압박 | **숨기지 말고 밀어올리기** — Postgres만 쓰면 Postgres를 드러내라 |
| 추상 타입 인터페이스 하나만 구현 존재 | **추상화 연기 (YAGNI)** — 두 번째 구현이 생길 때까지 구체 타입 |
| 라이브러리 교체 가능성을 위한 래퍼 | **실제 교체할 확률 vs 현재 비용** — 보통 비용이 더 큼 |
| 오류 전파 | **도메인 에러로 변환** — 하위 라이브러리 예외 그대로 던지지 않기 |

## 좋은 추상화의 조건

1. **투명성**: 필요하면 내부를 볼 수 있다 (로깅, 디버거, explain)
2. **예측 가능성**: 성능·실패 모드가 예측 가능하다
3. **도메인 정합성**: 추상화의 언어가 해결하려는 도메인 언어다 (하위 구현 언어가 아니다)
4. **비용의 명시성**: 추상화를 씀으로써 감수하는 비용(성능, 학습)을 사용자가 안다
5. **명시적 escape**: 추상을 벗어나는 공식 경로가 있다

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| ENCAPSULATION | 캡슐화는 "숨겨야 한다"는 규범, Leaky Abstraction은 "완전히 숨길 수는 없다"는 현실 |
| POLA | 누수가 사용자를 놀라게 하면 POLA도 위반 |
| HYRUMS-LAW | 누수된 동작은 Hyrum's Law로 사실상 계약이 된다 |
| ORTHOGONALITY | 누수가 심하면 위 레이어가 아래 레이어에 얽혀 직교성이 무너진다 |
| YAGNI | "나중에 바꿀 수도 있으니"로 만든 추상화는 보통 누수만 남기고 이득이 없다 |
| KISS | 얇고 정직한 추상화가 두껍고 누수되는 추상화보다 낫다 |

## 주의

- Leaky Abstraction ≠ "추상화를 쓰지 마라". 추상화는 필요하다, 다만 공짜가 아니다
- "내 추상화는 누수되지 않는다"는 자신 = 아직 엣지 케이스를 안 만났다는 뜻
- 누수는 **디버깅 가능성**으로 완화된다. 숨기되 드러낼 수 있게
- 사용자가 숙련자면 누수된 추상화도 견딜 수 있다. 초보자 상대 추상화일수록 누수가 치명적
- "추상화 비용 > 이득"이면 차라리 추상화를 걷어내는 것이 옳다
