# ORTHOGONALITY — 직교성

## 정의

> "두 컴포넌트가 직교한다 = 하나를 변경해도 다른 하나에 영향이 없다."

기하학에서 직교하는 두 축은 서로 독립적으로 움직인다.
소프트웨어에서도 마찬가지 — 직교적 시스템은 각 부분을 독립적으로 변경, 테스트, 교체할 수 있다.

## 핵심 판단

- **"A를 바꾸면 B도 바꿔야 하는가?"** — 그렇다면 A와 B는 직교하지 않는다
- **"이 컴포넌트를 제거해도 나머지가 동작하는가?"** — 동작하면 직교적
- **"이 변경의 영향 범위를 예측할 수 있는가?"** — 예측 불가면 직교성 부족

## SRP와의 관계

| | SRP | 직교성 |
|---|---|---|
| 관점 | 모듈 내부 | 모듈 간 관계 |
| 질문 | "변경 이유가 하나인가?" | "변경이 다른 곳에 전파되는가?" |
| 위반 결과 | 하나의 모듈이 여러 이유로 변경 | 하나의 변경이 여러 모듈에 전파 |

SRP를 지키면 직교성이 따라오는 경우가 많지만, SRP를 지켜도 직교하지 않을 수 있다 (예: 두 모듈이 각각 단일 책임이지만 공유 상태로 암묵적 결합).

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 임계값 | 검증 방법 |
|------|--------|-----------|
| 순환 의존 | 0개여야 함 | dependency-cruiser, import 그래프 분석 |
| 하나의 커밋이 수정하는 모듈 수 | > 3개 모듈 | `git log --stat` 분석 |
| 공유 전역 상태 (global, singleton) | 2곳+ 참조 | grep `global\|singleton\|shared_state` |
| 하나의 파일 변경 시 깨지는 테스트 파일 수 | > 2개 | 테스트 실행 후 실패 범위 분석 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| Shotgun Surgery | 하나의 변경을 위해 여러 파일을 수정 |
| 모듈 A의 내부 변경이 모듈 B의 테스트를 깨뜨림 | 암묵적 결합 |
| 기능 플래그가 여러 모듈에 분산 | 기능 경계가 모듈 경계와 불일치 |
| 설정 변경 하나가 예상치 못한 곳에 영향 | 설정의 직교성 부족 |
| "이거 건드리면 뭐가 깨질지 몰라" | 직교성 결여의 확실한 증거 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 순환 의존 0, 변경 영향 범위 예측 가능, 모듈 독립 테스트 가능 |
| **WARN** | 순환 의존 1-2개 OR 일부 변경이 2-3 모듈에 전파 |
| **FAIL** | 순환 의존 3개+ OR Shotgun Surgery 반복 OR 공유 상태로 암묵적 결합 |

## 직교성 확보 패턴

| 상황 | 패턴 |
|------|------|
| 모듈 간 암묵적 의존 | **명시적 인터페이스** — 계약을 인터페이스로 정의 |
| 공유 전역 상태 | **의존성 주입** — 공유 대신 주입 |
| 기능 변경이 여러 모듈에 전파 | **이벤트/메시지** — 직접 호출 대신 이벤트 발행 |
| 설정이 코드 곳곳에 산재 | **설정 집중화** — 한 곳에서 읽고 주입 |
| 순환 의존 | **의존성 역전** — 인터페이스로 방향 끊기 |
| DB 스키마 변경이 여러 서비스에 전파 | **Anti-Corruption Layer** — 각 서비스가 자체 모델 유지 |

## 직교성 테스트

프로젝트의 직교성을 빠르게 점검하는 질문:

1. 이 모듈의 단위 테스트가 다른 모듈의 구현에 의존하는가?
2. 하나의 기능을 끄려면 몇 곳을 수정해야 하는가?
3. 새 팀원이 하나의 모듈만 이해하고 그 모듈을 수정할 수 있는가?
4. 라이브러리/프레임워크를 교체하면 비즈니스 로직도 변경해야 하는가?

## 적용 범위

직교성은 코드에만 적용되는 것이 아니다:

| 영역 | 직교적 | 비직교적 |
|------|--------|----------|
| 코드 | 모듈 독립 변경 | Shotgun Surgery |
| 팀 | 팀 간 독립 배포 | 릴리스 동기화 필요 |
| 인프라 | 서비스 독립 스케일링 | 전체 재배포 |
| 스킬/도구 | 각 스킬이 독립 작동 | 스킬 A가 스킬 B에 의존 |
| 설정 | 환경 변수 독립 | 하나 바꾸면 다른 것도 조정 |

## 주의

- 100% 직교는 불가능하고 불필요하다. 핵심은 **불필요한 결합을 제거**하는 것
- 과도한 분리는 간접 참조 폭발 → KISS와 균형
- 직교성을 위해 코드를 복사(중복)하는 것은 DRY 위반 — 인터페이스로 해결
- 성능 최적화를 위해 의도적으로 결합하는 경우는 의도된 위반으로 기록
