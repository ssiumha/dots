# ROT — Rule of Three

## 정의

> "Three strikes and you refactor."

1번째: 그냥 작성한다.
2번째: 중복이 보이지만 참는다.
3번째: 이제 패턴이 보인다 — 추상화한다.

DRY가 "반복하지 마라"라면, Rule of Three는 **"언제 반복을 제거할지"**의 타이밍 규칙이다. 너무 빨리 추상화하면 잘못된 추상화가 되고, 너무 늦으면 중복이 퍼진다.

## 핵심 판단

- **"이 코드가 몇 번째 등장인가?"** — 2번까지는 허용, 3번부터 추상화 검토
- **"반복되는 부분이 정말 같은 이유로 존재하는가?"** — 형태가 같아도 이유가 다르면 별개
- **"추상화했을 때 세 곳 모두 자연스럽게 사용할 수 있는가?"** — 억지 추상화 경계

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 동일/유사 코드 블록 3개+ | 코드 클론 검출 (jscpd, PMD CPD) |
| 같은 패턴의 함수 3개+ (이름만 다름) | 함수 시그니처 유사도 분석 |
| 동일 조건 분기 3곳+ | grep 동일 조건문 패턴 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| 2번째 중복에서 성급하게 추상화 | 아직 패턴이 불확실한 상태에서 일반화 |
| 형태는 같지만 의미가 다른 코드를 합침 | "우연히 비슷한" 것을 DRY로 착각 |
| 추상화 후 조건 분기가 늘어남 | 잘못된 추상화 — 차라리 중복이 나음 |
| 4번째, 5번째 복붙이 계속 발생 | 3번째에서 추상화를 놓침 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 3회 이상 반복 코드 없음 OR 반복이 있으면 공통 추상화 존재 |
| **WARN** | 3회 반복 코드 1-2곳 OR 2회 반복에서 성급한 추상화 |
| **FAIL** | 3회+ 반복 코드 3곳 이상 OR 잘못된 추상화로 조건 분기 폭발 |

## 올바른 추상화 vs 잘못된 추상화

```
// 3곳에서 반복 — 추상화 타이밍

// ❌ 잘못된 추상화: 차이를 플래그로 처리
function sendNotification(type, user, data) {
  if (type === 'email') { ... }
  else if (type === 'slack') { ... }
  else if (type === 'sms') { ... }
}

// ✅ 올바른 추상화: 공통 인터페이스 + 개별 구현
interface Notifier { send(user, data) }
class EmailNotifier implements Notifier { ... }
class SlackNotifier implements Notifier { ... }
class SMSNotifier   implements Notifier { ... }
```

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 동일 코드 3회 반복 | **Extract Function/Method** — 공통 로직 추출 |
| 유사 구조 3회 (부분만 다름) | **Template Method / Strategy** — 변하는 부분만 분리 |
| 같은 데이터 변환 3곳 | **공유 유틸 / 파이프라인** — 변환 로직 일원화 |
| 2회 반복인데 추상화하고 싶은 충동 | **참는다** — TODO 코멘트만 남기고 3번째를 기다린다 |

## DRY와의 관계

| | DRY | Rule of Three |
|--|-----|---------------|
| 질문 | "반복이 있는가?" | "언제 반복을 제거할 것인가?" |
| 위험 | 적용 안 함 → 중복 확산 | 너무 빨리 적용 → 잘못된 추상화 |
| 보완 | ROT가 타이밍을 잡아줌 | DRY가 목표를 잡아줌 |

## 주의

- "3"은 정확한 숫자가 아니라 **"충분한 사례가 모이면"**이라는 휴리스틱
- 형태가 같아도 변경 이유가 다르면 별개다. 세금 계산과 할인 계산이 우연히 같은 공식이어도 합치면 안 된다
- 추상화는 되돌리기 어렵다. 잘못된 추상화보다 중복이 낫다 (Sandi Metz: "duplication is far cheaper than the wrong abstraction")
- 테스트 코드에서의 반복은 가독성을 위해 허용되는 경우가 많다
