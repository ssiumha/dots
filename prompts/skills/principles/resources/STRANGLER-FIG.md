# STRANGLER-FIG — Strangler Fig Pattern

## 정의

> "Gradually create a new system around the edges of the old, letting it grow slowly over several years until the old system is strangled."

교살 무화과나무는 숙주 나무를 감싸면서 자라다가 결국 숙주를 대체한다. 소프트웨어에서도 레거시를 한 번에 버리고 새로 만드는 대신, 새 시스템이 기존 시스템을 점진적으로 감싸면서 기능을 하나씩 가져온다.

## 핵심 판단

- **"이 교체를 한 번에 할 수 있는가?"** — 못하면 Strangler Fig 적용
- **"교체 중에도 기존 시스템이 작동해야 하는가?"** — 그렇다면 반드시 Strangler Fig
- **"Big Bang 교체의 리스크를 감당할 수 있는가?"** — 못하면 점진적으로

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| "v2" 브랜치가 6개월+ 분리되어 개발 중 | 브랜치 생성일 vs 현재 날짜 |
| 새 시스템과 기존 시스템 사이 라우팅/프록시 없음 | 인프라 설정에서 프록시/라우팅 부재 |
| 마이그레이션 중간 상태가 없음 (전부 old 또는 전부 new) | feature flag / 라우팅 규칙 부재 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "전부 새로 만들자" | Big Bang 교체 시도 |
| "마이그레이션 3개월이면 될 거야" (대규모) | 과소 추정 |
| 교체 중 기존 시스템 유지보수 중단 | 이중 운영 거부 |
| 새 시스템이 완성될 때까지 릴리즈 불가 | 가치 전달 중단 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | 점진적 교체, 각 단계가 독립 배포 가능, 롤백 가능, 기존 시스템 계속 작동 |
| **WARN** | 교체 계획은 있으나 단계가 크거나, 롤백 전략 미흡 |
| **FAIL** | Big Bang 교체 시도 OR 6개월+ 분리 브랜치 OR 교체 중 서비스 중단 |

## Strangler Fig 실행 단계

```
1. Intercept (가로채기)
   → 기존 시스템 앞에 프록시/라우터 배치
   → 모든 트래픽이 프록시를 통과

2. Route (라우팅)
   → 새 기능은 새 시스템으로 라우팅
   → 기존 기능은 여전히 기존 시스템으로

3. Migrate (이전)
   → 기존 기능을 하나씩 새 시스템으로 이전
   → 이전 완료된 기능의 라우팅 전환

4. Retire (폐기)
   → 기존 시스템에 트래픽이 없으면 제거
   → 또는 읽기 전용으로 유지 (데이터 참조)
```

## 적용 영역

| 영역 | 적용 방법 |
|------|-----------|
| **API 버전 교체** | v1 → v2: 프록시에서 엔드포인트별 라우팅, 점진적 v2 전환 |
| **프레임워크 교체** | Rails → Next.js: 페이지별 전환, nginx에서 라우팅 |
| **DB 마이그레이션** | 이중 쓰기 → 새 DB 읽기 전환 → 구 DB 폐기 |
| **모놀리스 분해** | 기능별 서비스 추출, API Gateway에서 라우팅 |
| **라이브러리 교체** | 어댑터 패턴으로 감싸기 → 내부 구현 점진 교체 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| Big Bang 교체 계획 | **기능 단위 분할** — 가장 독립적인 기능부터 이전 |
| 이전 중 양쪽 동기화 문제 | **이중 쓰기 + 비교 검증** — 양쪽에 쓰고 결과 비교 |
| 롤백 불가 구조 | **Feature Flag** — 라우팅을 코드 레벨에서 제어 |
| 데이터 마이그레이션 부담 | **점진적 마이그레이션** — 접근되는 데이터만 on-demand 이전 |
| "새로 만드는 게 빠르다" 주장 | **Gall's Law 상기** — 작동하는 시스템에서 진화하라 |

## Anti-Corruption Layer와의 관계

Strangler Fig 실행 중 새 시스템과 기존 시스템 사이에는 **Anti-Corruption Layer(ACL)**이 필요하다:

- 기존 시스템의 모델이 새 시스템을 오염시키지 않도록 변환
- ACL이 Strangler의 "껍질" 역할
- 이전 완료 시 ACL도 함께 제거

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| GALLS-LAW | Strangler Fig는 Gall's Law의 실천 전략 — 작동하는 시스템에서 점진적으로 |
| BC | 모놀리스를 BC 단위로 분해할 때 Strangler Fig 적용 |
| CHESTERTONS-FENCE | 기존 시스템의 각 부분을 이전하기 전 "왜 이렇게 되어 있는지" 이해 필요 |
| BROKEN-WINDOWS | Big Bang 교체 실패 → 반만 된 새 시스템 + 방치된 구 시스템 = 최악의 깨진 유리창 |

## 주의

- Strangler Fig는 느리다. 이중 운영 비용이 있다. 하지만 Big Bang 실패보다 싸다
- 모든 교체에 Strangler Fig가 필요한 건 아니다. 영향 범위가 작으면 직접 교체해도 된다
- 이중 운영 기간을 최소화하려면 **가장 트래픽이 많은 기능부터** 이전
- "Strangler Fig 중입니다"가 영원히 계속되면 안 된다. 완료 기한과 폐기 계획 필수
- 데이터 마이그레이션이 가장 어렵다. 코드 교체보다 데이터 동기화가 핵심 난이도
