# YAGNI — You Aren't Gonna Need It

## 정의

> "항상 실제로 필요할 때 구현하라, 필요할 것이라고 예상할 때가 아니라."

예측은 거의 항상 틀린다. 미래를 위한 코드는 현재의 복잡도를 높이고, 정작 그 미래가 오지 않는다.

## 핵심 판단

- **"지금 요구사항에 이게 필요한가?"** — 아니면 만들지 않는다
- **"이걸 누가 쓰는가?"** — 호출자가 없으면 필요 없다
- **"나중에 필요하면?"** — 그때 만들면 된다. 그때가 더 잘 만든다

## 위반 신호

### 기계적 검증 (자동화 가능)

| 신호 | 검증 방법 |
|------|-----------|
| 사용되지 않는 함수/메서드 | 정적 분석, dead code 탐지 |
| 사용되지 않는 파라미터/플래그 | ast-grep, lint (unused parameter) |
| 사용되지 않는 설정 옵션 | grep으로 참조 확인 |
| 빈 인터페이스 구현 (NotImplementedError) | grep `NotImplementedError\|TODO\|pass$` |
| 단일 구현체만 있는 인터페이스/추상 클래스 | 구현체 수 카운트 |

### 판단 필요 (사람/LLM)

| 신호 | 설명 |
|------|------|
| "나중에 쓸 수도 있으니까" | 미래 예측 기반 구현의 전형적 신호 |
| 과도한 설정 가능성 (configurability) | 변경되지 않을 것을 설정으로 빼놓음 |
| 미래 확장을 위한 추상화 계층 | 지금은 하나뿐인 구현체에 인터페이스 |
| 주석의 "// for future use" | 미래를 위한 코드의 명시적 증거 |
| 빈 훅/콜백 포인트 | 확장점은 있으나 사용자가 없음 |

## 스코어링

| 등급 | 기준 |
|------|------|
| **PASS** | dead code 0, 모든 코드가 현재 요구사항에 직결 |
| **WARN** | dead code 1-3개 OR 단일 구현체 인터페이스 1-2개 |
| **FAIL** | dead code 4개+ OR "for future use" 패턴 반복 OR 미사용 설정 다수 |

## 개선 패턴

| 상황 | 적용 |
|------|------|
| 미사용 코드 | **삭제** — git에 있으니 필요하면 되살린다 |
| 단일 구현 인터페이스 | **Inline Interface** — 구현이 2개 될 때 추출 |
| 과도한 설정 | **하드코딩** — 변경 필요 시 그때 설정으로 추출 |
| 미래 확장점 | **제거** — 확장이 실제로 필요할 때 추가 |

## 다른 원칙과의 관계

| 원칙 | 관계 |
|------|------|
| [[KISS]] | YAGNI는 기능 측면, KISS는 구조 측면 — 같은 단순함의 두 축 |
| [[PREMATURE-OPTIMIZATION]] | YAGNI의 성능 영역 특화 — "필요할 때까지 하지 마라"의 성능 변형 |
| [[OCP]] | 긴장 관계 — OCP는 확장점 미리 설계, YAGNI는 필요할 때. 실제 변경 2회+ 발생한 축에만 OCP |
| [[ROT]] | YAGNI의 실용 적용 — 3번째 반복 전에는 추상화 보류 |
| [[TECHNICAL-DEBT]] | YAGNI를 따르면 의도적 부채 감소 — 불필요한 기능에 부채 안 짊 |

## 주의

- YAGNI ≠ 설계 포기. 확장 가능한 구조는 유지하되, 구현은 현재 필요한 것만
- YAGNI와 OCP는 긴장 관계: OCP는 확장점을 미리 설계하라 하고, YAGNI는 필요할 때 하라 한다. 해결: 실제 변경이 2회+ 발생한 축에만 OCP 적용
- 라이브러리/프레임워크 코드는 예외 — 범용성이 요구사항 자체
