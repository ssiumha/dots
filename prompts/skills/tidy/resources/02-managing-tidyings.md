# Managing Tidyings (Part II)

Kent Beck "Tidy First?" Part II — 정리를 **언제, 얼마나** 할 것인가.

## Separate Tidying

구조 변경(S)과 행동 변경(B)은 **반드시 별도 커밋/PR**로 분리한다.

- 같은 커밋에 섞으면: 리뷰 어려움, revert 위험, 원인 추적 불가
- S 커밋은 테스트 결과 불변, B 커밋은 테스트 결과 변경
- 가장 중요한 규율: **하나의 변경에 S와 B를 섞지 않는다**

## Chaining

작은 정리를 순차적으로 연결한다.

```
Guard Clause → Dead Code 제거 → Explaining Variable → 본 작업
```

- 각 단계가 다음 단계를 자연스럽게 드러냄
- 하나의 큰 리팩토링이 아닌, 연결된 작은 정리의 흐름
- 각 단계마다 커밋 → 언제든 멈출 수 있음

## Batch Sizes

**당장 필요한 범위만** 정리한다.

- 파레토 분포: 소수의 정리가 대부분의 가치를 제공
- "이왕 하는 김에" 는 범위 확산의 신호
- 정리 대상을 발견해도, 지금 하는 작업과 무관하면 **Later** 목록으로

## Rhythm

**1시간 규칙**: 정리에 1시간 이상 빠지면 멈추고 재평가한다.

- 더 작은 최소 변경을 놓쳤을 가능성이 높음
- 정리 → 구현 → 정리 → 구현 리듬 유지
- 정리의 만족감에 빠지지 않기 — 목적은 행동 변경을 쉽게 만드는 것

## Getting Untangled

정리와 행동 변경이 엉켰을 때 3가지 선택지:

| 선택지 | 방법 | 적합한 상황 |
|--------|------|-------------|
| **되감기** | 변경 취소 후 S/B 분리하여 재작업 | 변경량 적을 때 |
| **앞으로 가기** | 현재 상태에서 S만 추출하여 별도 커밋 | 변경량 많지만 분리 가능할 때 |
| **그냥 가기** | 섞인 채로 커밋 (비추천) | 긴급하거나 분리 비용 > 가치일 때 |

## First, After, Later, Never

정리 **타이밍** 판단 프레임워크:

### First (먼저 정리)

다음 행동 변경이 정리하면 **확실히** 쉬워질 때.

- 복잡한 조건문 안에 기능을 추가해야 할 때 → Guard Clause 먼저
- 레거시 API를 호출하는 코드를 수정해야 할 때 → New Interface 먼저
- **판단 기준**: 정리 비용 < 정리 없이 변경하는 추가 비용

### After (변경 후 정리)

행동 변경을 완료한 뒤, 구조 개선 기회가 보일 때.

- 기능 구현 중 발견한 중복 → 기능 커밋 후 Extract Helper
- 동작하는 코드를 더 깔끔하게 → 별도 정리 커밋
- **주의**: 정리는 별도 커밋. 기능 커밋에 섞지 않는다

### Later (나중에)

지금 하는 작업과 직접 관련 없는 정리.

- 발견만 하고 목록에 기록
- 관련 작업이 생길 때 First로 승격
- 영원히 Later에 머물 수도 있음 (그것도 괜찮다)

### Never (하지 않음)

정리하지 않는 것이 올바른 선택인 경우:

- 코드가 안정적이고 변경 예정 없음
- 정리 비용 > 기대 가치
- 곧 폐기/교체될 코드
- 정리가 위험을 수반 (깊은 결합, 이해 부족)
